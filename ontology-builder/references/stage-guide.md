# 五阶段引导问题详细指南

本文档包含5个阶段的详细引导问题、示例和处理逻辑。

---

## 阶段一：理解业务对象

### 引导问题（逐个提问，等待回答）

**Q1: 对象识别**
```
你想管理什么业务对象？
例如：供应商、客户、订单、设备、产品、员工等
```

**Q2: 业务场景**
```
你的团队为什么需要管理这个对象？主要解决什么问题？
例如：
- 追踪设备运行状态，及时发现故障
- 管理供应商信息，评估供应风险
- 监控订单进度，提升交付效率
```

**Q3: 关键人员**
```
哪些人会使用这个对象？他们的角色是什么？
例如：
- 运维工程师（查看设备状态）
- 采购经理（评估供应商）
- 客服专员（处理订单问题）
```

### 检查点话术

```
✅ 我理解了：
- 业务对象：【对象名称】
- 业务场景：【场景描述】
- 使用人员：【角色列表】

这个理解正确吗？有需要补充或修改的地方吗？
```

## 阶段二：定义数据字段

### 引导逻辑

1. 加载 `references/property-guide.md`
2. 告诉用户："现在我们来确定【对象名称】需要记录哪些信息。我会帮你分类整理，不用担心技术细节。"
3. 逐个提问Q4-Q6
4. 分类整理：固定属性、变化属性、时序属性
5. 展示清单，等待确认

### 引导问题

**Q4: 基础信息**
```
首先，【对象名称】有哪些基本信息需要记录？
例如设备对象：
- 设备编号（如 EQ-001）
- 设备名称（如 离心机）
- 所属车间（如 A栋2楼）
- 安装日期
```

**Q5: 状态信息**
```
【对象名称】有哪些状态需要追踪？这些状态会变化吗？
例如设备对象：
- 运行状态（运行中/停机/维修中）
- 实时温度（会随时间变化）
- 累计运行小时数
```

**Q6: 历史追溯**
```
哪些数据需要查看历史变化？
例如：
- ✅ 需要：设备温度（查看24小时趋势）
- ❌ 不需要：设备编号（固定不变）
```

### 检查点话术

```
✅ 我整理了【对象名称】的数据字段：

**固定信息**
- 字段1：【名称】【类型】【说明】
- 字段2：...

**变化信息**
- 字段3：【名称】【类型】【说明】

**历史追踪**
- 字段4：【名称】【类型】【说明】

这些字段完整吗？有遗漏或需要调整的吗？
```

## 阶段三：定义计算规则

### 引导逻辑

1. 加载 `references/function-guide.md`
2. 告诉用户："有些信息可以自动计算，不需要人工填写。我们来看看【对象名称】需要哪些自动判断或计算。"
3. 逐个提问Q7-Q9
4. 分类整理：风险判断、状态推断、业务计算
5. 展示清单，等待确认

### 引导问题

**Q7: 风险判断**
```
对于【对象名称】，有哪些情况需要系统自动判断风险或异常？
例如设备对象：
- 温度超过90°C → 高风险
- 连续运行超过72小时 → 需要保养
- 故障率超过5% → 供应商需要改进
```

**Q8: 状态推断**
```
有哪些状态可以根据其他数据自动推断？
例如：
- 根据最后心跳时间推断"设备是否在线"
- 根据订单日期推断"是否逾期"
- 根据库存数量推断"是否需要补货"
```

**Q9: 业务计算**
```
有哪些指标需要自动计算？
例如：
- 设备利用率 = 运行时间 / 总时间
- 供应商评分 = 质量分×0.5 + 交付分×0.3 + 价格分×0.2
- 订单完成率 = 已完成订单数 / 总订单数
```

### 检查点话术

```
✅ 我整理了【对象名称】的自动计算规则：

**风险判断规则**
1. 规则1：【条件】→【结果】
2. 规则2：...

**状态推断规则**
1. 规则1：【依据】→【推断结果】

**业务计算公式**
1. 计算1：【公式】【说明】

这些规则符合你的业务需求吗？
```

## 阶段四：定义操作按钮

### 引导逻辑

1. 加载 `references/action-guide.md`
2. 告诉用户："现在我们来设计用户可以对【对象名称】做什么操作。这些操作会变成界面上的按钮或表单。"
3. 逐个提问Q10-Q13
4. 分类整理：创建、更新、删除、业务操作
5. 展示清单，等待确认

### 引导问题

**Q10: 创建操作**
```
用户需要创建新的【对象名称】吗？创建时需要填写哪些必填信息？
例如创建新设备：
- 必填：设备编号、设备类型、所属车间
- 选填：负责人、保修期
```

**Q11: 更新操作**
```
用户可以修改【对象名称】的哪些信息？谁有权限修改？
例如：
- 运维人员可以更新"设备状态"
- 管理员可以修改"所属车间"
- 普通用户只能查看，不能修改
```

**Q12: 关键业务操作**
```
除了基本的增删改查，【对象名称】还需要哪些特殊操作？
例如设备对象：
- "紧急停机"操作（立即断电）
- "申请报废"操作（提交审批）
- "预约保养"操作（创建工单）
```

**Q13: 权限控制**
```
对于每个操作，哪些角色可以执行？有特殊限制吗？
例如：
- "紧急停机"：只有高级技术员在工作时间可执行
- "申请报废"：设备使用年限>5年才能申请
- "修改供应商信息"：只有采购经理可以操作
```

### 检查点话术

```
✅ 我整理了【对象名称】的操作列表：

**创建操作**
- 操作名称：【名称】
- 必填字段：【字段列表】
- 权限：【谁可以执行】

**更新操作**
- 操作1：【名称】【可修改的字段】【权限】
- 操作2：...

**业务操作**
- 操作1：【名称】【触发条件】【权限】【执行效果】

这些操作覆盖了你的业务需求吗？
```

## 阶段五：定义自动化规则

### 引导逻辑

1. 加载 `references/automation-guide.md`
2. 告诉用户："最后，我们来设置一些'自动化规则'。当满足特定条件时，系统会自动执行操作，不需要人工干预。"
3. 逐个提问Q14-Q16
4. 分类整理：异常响应、流程编排、定时任务
5. 展示清单，等待确认

### 引导问题

**Q14: 异常响应**
```
当【对象名称】出现异常情况时，系统应该自动做什么？
例如设备对象：
- 当温度超过90°C时 → 自动执行"紧急停机"操作
- 当故障率>10%时 → 自动发送邮件给设备经理
- 当离线超过5分钟时 → 自动发送告警到企业微信
```

**Q15: 流程编排**
```
某个操作完成后，需要自动触发其他操作吗？
例如：
- 当"申请报废"提交后 → 自动创建审批工单 + 通知审批人
- 当"设备停机"执行后 → 自动记录停机日志 + 计算停机损失
- 当"供应商评分"更新后 → 自动更新"供应商等级"字段
```

**Q16: 定时任务**
```
有需要定期自动执行的任务吗？
例如：
- 每天早上8点 → 自动生成设备健康报告
- 每周一 → 自动检查过期的保养计划
- 每月1号 → 自动统计上月设备故障率
```

### 检查点话术

```
✅ 我整理了【对象名称】的自动化规则：

**异常响应规则**
1. 触发条件：【条件】
   自动执行：【操作】
   通知方式：【邮件/短信/企业微信】

**流程编排规则**
1. 当【操作A】完成后 → 自动执行【操作B】→ 再执行【操作C】

**定时任务**
1. 时间：【频率】
   任务：【自动执行的操作】

这些自动化规则合理吗？会不会有误报或过度干预？
```

## 使用建议

### 何时使用本指南

- 在对应阶段需要详细的引导问题时
- 需要参考检查点话术时
- 需要回顾某个阶段的逻辑时

### 关键原则

- **逐个提问**：不要一次性问所有问题
- **等待回答**：每个问题都要等用户回答后再继续
- **提供例子**：每个问题都有2-3个具体例子
- **检查点确认**：每个阶段结束都要明确确认

### 与主SKILL.md的关系

- SKILL.md：核心流程、原则、anti-rationalization
- 本文件：详细的引导问题和话术
- Property/Function/Action/Automation-guide.md：概念和技术细节
