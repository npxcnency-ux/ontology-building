# 五阶段引导问题详细指南

本文档包含5个阶段的详细引导问题、示例和处理逻辑。

---

## 阶段一：理解业务场景与对象全景

### 引导问题（逐个提问，等待回答）

**Q1: 业务场景**
```
请描述一下你想要管理的业务场景。
你的团队在日常工作中遇到什么问题，希望通过系统来解决？

例如：
- "我们车间有很多设备，温度异常了没人知道，保养也经常漏掉"
- "供应商太多了，谁好谁差全靠感觉，想做个评估体系"
- "客户的维修工单全靠Excel管，经常丢单"
```

**Q2: 关键人员**
```
哪些人会使用这个系统？他们的角色是什么？
例如：
- 运维工程师（查看设备状态）
- 采购经理（评估供应商）
- 客服专员（处理订单问题）
```

**Q3: 对象全景分析**

基于Q1-Q2的回答，Agent自行分析并向用户展示：

```
根据你描述的业务场景，我识别出以下业务对象和它们之间的关系：

📊 对象全景图：

  ┌────────┐  属于   ┌────────┐  产生   ┌────────┐
  │  车间   │←──────│  设备   │──────→│  工单   │
  └────────┘       └────────┘       └────────┘
                                       │
                                  分配给 │
                                       ↓
                                   ┌────────┐
                                   │  技师   │
                                   └────────┘

涉及约 4 个业务对象。

你觉得这个分析完整吗？
- 如果有遗漏的"东西"或关系，请告诉我，我来更新全景图
- 如果觉得没问题，我们就基于这个全景图继续
```

**注意**：
- 全景图基于用户描述自动生成，不同场景图不同
- 使用简易ASCII图或项目符号列表呈现
- 对象之间的关系用业务动词描述（属于、产生、分配给...）
- **用户可以补充或修改**：如果用户新增了对象或关系，Agent更新全景图后再次确认
- **用户也可以直接接受**：确认后进入Q4选择核心对象

**Q4: 核心对象选择**
```
这些对象中，哪个是你最急需管理的核心对象？
我建议从它开始建模，完成后再逐步扩展到其他对象。

建议的建模顺序：
1. 设备（核心，本次建模）
2. 工单（与设备强关联，建议第二个）
3. 技师（配合工单管理）
4. 车间（相对独立，可以后续补充）

你同意这个顺序吗？或者你有不同的优先级？
```

### 检查点话术

```
✅ 我理解了你的业务场景：
- 业务场景：【场景描述】
- 核心痛点：【痛点描述】
- 使用人员：【角色列表】

📊 对象全景图：
  【对象A（核心）】←关系→【对象B】←关系→【对象C】
  共涉及 N 个业务对象。

🎯 建模计划：
  1. 先建模：【核心对象】（本次完成）
  2. 后续迭代：【对象B】→【对象C】→ ...

这个理解正确吗？有需要补充或修改的地方吗？
```

## 阶段二：定义数据字段与对象关联

### 引导逻辑

1. **回顾上一阶段**：用1-2句话总结阶段一的结论（"我们确定了N个业务对象，选择【核心对象】作为本次建模重点。现在来确定它需要记录哪些信息。"）
2. 加载 `references/property-guide.md`
3. 告诉用户："现在我们来确定【核心对象名称】需要记录哪些信息。我会帮你分类整理，不用担心技术细节。"
4. 逐个提问Q5-Q7
5. 分类整理：固定属性、变化属性、时序属性
6. 追问数据来源
7. 加载 `references/links-guide.md`，确认对象关联关系
8. 展示字段清单和关联关系清单，等待确认

### 引导问题

**Q5: 基础信息**
```
首先，【核心对象名称】有哪些基本信息需要记录？
例如设备对象：
- 设备编号（如 EQ-001）
- 设备名称（如 离心机）
- 所属车间（如 A栋2楼）
- 安装日期
```

**Q5b: 唯一标识**（Q5回答后立即追问，不可跳过）
```
在你刚才列出的这些信息中，哪一个可以唯一标识每一个【核心对象名称】？
也就是说，不会有两个【核心对象名称】共用同一个值的字段。

例如：
- 设备编号（每台设备编号唯一，如 EQ-001）
- 工单号（每个工单号唯一，如 WO-20240318-001）
- 身份证号（每个员工唯一）

💡 如果现有字段中没有天然唯一的，我们可以设计一个编号规则来生成。
```

**Q6: 状态信息**
```
【核心对象名称】有哪些状态需要追踪？这些状态会变化吗？
例如设备对象：
- 运行状态（运行中/停机/维修中）
- 实时温度（会随时间变化）
- 累计运行小时数
```

**Q7: 历史追溯**
```
哪些数据需要查看历史变化？
例如：
- ✅ 需要：设备温度（查看24小时趋势）
- ❌ 不需要：设备编号（固定不变）

对于需要查看历史的字段，历史数据大概要保留多久？
- 30-90天：适合实时传感器数据（温度、压力），主要看近期趋势
- 1-2年：适合状态变更历史、业务指标，用于年度对比
- 按法规要求（3-7年）：医疗、金融等行业可能有合规要求

💡 如果不确定，先按90天开始，后续可以调整。
过期数据可以归档到低成本存储，不会丢失。
```

**Q7b: 数据来源**（Q7确认后立即追问，不可跳过）
```
对于我们刚才讨论的每个字段，数据从哪里来？

- 人工录入：由用户在界面上填写（如设备名称、负责人、备注）
- 外部系统同步：从其他系统自动导入（如从SAP同步设备编号，从IoT平台同步温度）
- 系统自动生成：系统自动产生（如创建时间、自增编号）

对于从外部系统同步的字段，还需要知道：
1. 从哪个系统来？（如 SAP、MES、GPS平台、油卡系统）
2. 多久同步一次？（实时推送、每5分钟、每小时、每天）

💡 不确定具体的系统名称也没关系，先标注"来自XX类系统"，
具体细节可以后续和IT团队确认。
```

**Q8: 对象关联关系**

基于阶段一的对象全景图，确认当前核心对象与其他对象的关系：

```
根据刚才的对象全景图，【核心对象名称】和以下对象有关系：

- 【核心对象】→【对象B】：一对多
  例如：一台设备可以产生多个工单
- 【核心对象】←【对象C】：多对一
  例如：多台设备属于同一个车间

这些关系对吗？还有遗漏的吗？

💡 判断技巧：如果你刚才想把"所属车间"记录为一个文本字段，
但"车间"本身也是一个需要独立管理的对象（有自己的编号、位置、负责人），
那它应该是一个关联关系而不是一个文本字段。
```

**Q8b: 级联行为**（Q8确认后立即追问，不可跳过）

确认每个关联关系的级联行为：

```
对于刚才确认的每个关联关系，如果主对象被删除了，关联的对象怎么办？

- 一起删除？（如：订单删除 → 订单明细一起删除）
- 清空关联？（如：员工离职 → 工单的负责人清空）
- 阻止删除？（如：供应商有进行中的订单 → 不能删除供应商）

💡 大多数情况下选"阻止删除"最安全——防止误删导致数据丢失。
如果不确定，先选"阻止删除"，后续可以调整。
```

### 检查点话术

```
✅ 我整理了【核心对象名称】的数据字段和关联关系：

**固定信息**
| 字段名 | 类型 | 必填 | 数据来源 | 说明 |
|--------|------|------|---------|------|
| 字段1  | 文本 | ✅必填 | 人工录入 | ... |
| 字段2  | 文本 | 选填 | 外部同步(SAP) | ... |

**变化信息**
| 字段名 | 类型 | 必填 | 数据来源 | 说明 |
|--------|------|------|---------|------|
| 字段3  | 数字 | ✅必填 | 传感器同步 | ... |

**历史追踪**
| 字段名 | 类型 | 必填 | 数据来源 | 说明 |
|--------|------|------|---------|------|
| 字段4  | 数字 | 选填 | 外部同步 | ... |

🔑 唯一标识（主键）：【字段名】

**对象关联关系**
- 【核心对象】→ 产生 →【工单】（一对多，级联：阻止删除）
- 【核心对象】← 属于 ←【车间】（多对一，级联：清空关联）

请确认：
1. 每个字段的"必填/选填"标记是否正确？（建议必填字段≤7个）
2. 字段和关联关系是否完整？有遗漏或需要调整的吗？
```

## 阶段三：定义计算规则

### 引导逻辑

1. **回顾上一阶段**：简述阶段二的结论（"我们为【核心对象】定义了X个数据字段和Y个关联关系。现在来看哪些信息可以自动计算。"）
2. 加载 `references/function-guide.md`
3. 告诉用户："有些信息可以自动计算，不需要人工填写。我们来看看【核心对象名称】需要哪些自动判断或计算。"
4. 逐个提问Q9-Q11
5. 分类整理：风险判断、状态推断、业务计算
6. 展示清单，等待确认

### 引导问题

**Q9: 风险判断**
```
对于【核心对象名称】，有哪些情况需要系统自动判断风险或异常？
例如设备对象：
- 温度超过90°C → 高风险
- 连续运行超过72小时 → 需要保养
- 故障率超过5% → 供应商需要改进
```

**Q10: 状态推断**
```
有哪些状态可以根据其他数据自动推断？
例如：
- 根据最后心跳时间推断"设备是否在线"
- 根据订单日期推断"是否逾期"
- 根据库存数量推断"是否需要补货"
```

**Q11: 业务计算**
```
有哪些指标需要自动计算？
例如：
- 设备利用率 = 运行时间 / 总时间
- 供应商评分 = 质量分×0.5 + 交付分×0.3 + 价格分×0.2
- 订单完成率 = 已完成订单数 / 总订单数
```

### 检查点话术

```
✅ 我整理了【核心对象名称】的自动计算规则：

**风险判断规则**
1. 规则1：【条件】→【结果】
2. 规则2：...

**状态推断规则**
1. 规则1：【依据】→【推断结果】

**业务计算公式**
1. 计算1：【公式】【说明】

这些规则符合你的业务需求吗？
```

## 阶段四：定义操作按钮

### 引导逻辑

1. **回顾上一阶段**：简述阶段三的结论（"我们定义了X个计算规则，包括风险判断、状态推断等。现在来设计用户可以执行的操作。"）
2. 加载 `references/action-guide.md`
3. 告诉用户："现在我们来设计用户可以对【核心对象名称】做什么操作。这些操作会变成界面上的按钮或表单。"
4. 逐个提问Q12-Q16
5. 分类整理：创建、更新、删除、业务操作
6. **追问回写需求**：哪些操作执行后需要同步数据到外部系统？
7. 展示操作清单（含回写配置），等待确认

### 引导问题

**Q12: 创建操作**
```
用户需要创建新的【核心对象名称】吗？创建时需要填写哪些必填信息？
例如创建新设备：
- 必填：设备编号、设备类型、所属车间
- 选填：负责人、保修期
```

**Q13: 更新操作**
```
用户可以修改【核心对象名称】的哪些信息？谁有权限修改？
例如：
- 运维人员可以更新"设备状态"
- 管理员可以修改"所属车间"
- 普通用户只能查看，不能修改
```

**Q14: 关键业务操作**
```
除了基本的增删改查，【核心对象名称】还需要哪些特殊操作？
例如设备对象：
- "紧急停机"操作（立即断电）
- "申请报废"操作（提交审批）
- "预约保养"操作（创建工单）
```

**Q15: 权限控制**
```
对于每个操作，哪些角色可以执行？有特殊限制吗？
例如：
- "紧急停机"：只有高级技术员在工作时间可执行
- "申请报废"：设备使用年限>5年才能申请
- "修改供应商信息"：只有采购经理可以操作
```

**Q15b: 前置条件**（Q15确认后立即追问，不可跳过）
```
对于刚才定义的每个操作，在什么条件下才能执行？有没有"不满足就不能操作"的限制？

例如：
- "启动设备"：只有当设备状态="停机"时才能执行（已报废的不能启动）
- "关闭工单"：只有当工单状态="已完成"时才能关闭（进行中的不能直接关闭）
- "审批通过"：只有当所有必填信息都已填写时才能审批

💡 判断技巧：想象一下，如果有人在"错误的时间"点了这个按钮，
会发生什么不好的后果？那个"错误的时间"就是前置条件要排除的场景。

如果某个操作没有特殊限制（如"修改备注"），说"无限制"就好。
```

**Q16: 回写需求**
```
刚才定义的这些操作中，有哪些执行后需要把结果同步回你们现有的系统？
例如：
- 审批通过后 → 需要回写到SAP更新资产状态
- 工单关闭后 → 需要更新ERP中的维修记录
- 发货确认后 → 需要同步物流信息到WMS仓储系统

💡 判断技巧：如果这个操作的结果不仅在本系统使用，
还需要让其他系统"知道"，那就需要回写。

如果需要回写，请告诉我：
1. 哪个操作需要回写？
2. 回写到哪个系统？（如SAP、ERP、MES、WMS等）
3. 需要同步哪些信息？
```

**注意**：
- 如果用户说"没有回写需求"或"暂时不需要"，记录为"暂无回写"并继续
- 回写需求会影响Action的外部调用配置和Automation的失败处理策略

### 检查点话术

```
✅ 我整理了【核心对象名称】的操作列表：

**创建操作**
- 操作名称：【名称】
- 必填字段：【字段列表】
- 权限：【谁可以执行】

**更新操作**
- 操作1：【名称】【可修改的字段】【权限】
- 操作2：...

**业务操作**
- 操作1：【名称】【触发条件】【权限】【执行效果】

**回写配置**（如有）
- 操作1 → 回写到【目标系统】：同步【字段A、字段B】
- 操作2 → 暂无回写

这些操作覆盖了你的业务需求吗？
```

## 阶段五：定义自动化规则

### 引导逻辑

1. **回顾上一阶段**：简述阶段四的结论（"我们定义了X个操作按钮，包括创建、更新和业务操作。最后，来设置系统自动执行的规则。"）
2. 加载 `references/automation-guide.md`
3. 告诉用户："最后，我们来设置一些'自动化规则'。当满足特定条件时，系统会自动执行操作，不需要人工干预。"
4. 逐个提问Q17-Q19
5. 分类整理：异常响应、流程编排、定时任务
6. 展示清单，等待确认

### 引导问题

**Q17: 异常响应**
```
当【核心对象名称】出现异常情况时，系统应该自动做什么？
例如设备对象：
- 当温度超过90°C时 → 自动执行"紧急停机"操作
- 当故障率>10%时 → 自动发送邮件给设备经理
- 当离线超过5分钟时 → 自动发送告警到企业微信
```

**Q18: 流程编排**
```
某个操作完成后，需要自动触发其他操作吗？
例如：
- 当"申请报废"提交后 → 自动创建审批工单 + 通知审批人
- 当"设备停机"执行后 → 自动记录停机日志 + 计算停机损失
- 当"供应商评分"更新后 → 自动更新"供应商等级"字段
```

**Q19: 定时任务**
```
有需要定期自动执行的任务吗？
例如：
- 每天早上8点 → 自动生成设备健康报告
- 每周一 → 自动检查过期的保养计划
- 每月1号 → 自动统计上月设备故障率
```

**Q19b: 限流与失败处理**（Q17-Q19全部确认后统一追问，不可跳过）
```
对于刚才定义的每条自动化规则，我们需要确认两件事：

1️⃣ **频率限制**：同一个对象短时间内反复触发这条规则时，怎么处理？
   - 例如：同一辆车1小时内油耗异常10次 → 只提醒1次还是每次都提醒？
   - 建议对每条规则设一个"最多多久提醒1次"的限制

2️⃣ **失败处理**：如果这条规则执行失败了（比如通知系统宕机、外部系统无响应），怎么办？
   - 自动重试？（适合重要但不紧急的操作）
   - 通知管理员？（适合需要人工介入的场景）
   - 直接跳过？（适合非关键的提醒类操作）

💡 如果不确定，建议选"自动重试 + 通知管理员"最安全。
```

### 检查点话术

```
✅ 我整理了【核心对象名称】的自动化规则：

**异常响应规则**
| 规则 | 触发条件 | 自动执行 | 通知方式 | 频率限制 | 失败处理 |
|------|---------|---------|---------|---------|---------|
| 1    | 【条件】 | 【操作】 | 邮件/企业微信 | 每小时最多1次 | 自动重试+通知管理员 |

**流程编排规则**
```
