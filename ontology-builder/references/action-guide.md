# 操作按钮指南（Action Guide）

Action（操作）是用户与系统交互的唯一方式。每个Action对应界面上的一个按钮、一个表单、或一个菜单项。

## 核心原则

### 什么是Action？
Action是"改变数据的双手"：
- **唯一写入途径**：只有Action可以创建、修改、删除数据
- **事务性保证**：要么全部成功，要么全部回滚
- **权限可控**：可以精确控制谁在什么条件下可以执行

### Action的生命周期
```
用户点击按钮 → 填写表单 → 系统校验 → 执行操作 → 更新数据 → 记录日志
```

### Action的事务性保证
Action是**原子操作**——要么全部成功，要么全部回滚，不会出现"做了一半"的情况。

```
示例："紧急停机"Action包含3个步骤：
  Step 1: 发送停机指令
  Step 2: 更新设备状态 → "停机"
  Step 3: 记录停机日志

事务保证：
  ✅ 3个步骤全部成功 → 提交
  ❌ 任何一步失败 → 全部回滚（就像什么都没发生）

为什么重要？
  如果Step 2成功但Step 3失败，没有事务保证的话：
  → 设备状态已变"停机"，但没有日志记录
  → 没人知道为什么停的、谁操作的
  → 审计和排查时无据可查
```

**设计建议**：如果一个Action的步骤过多（>5步），考虑拆分：
- **Action本身**：只做核心的数据写入（1-3步，必须原子性保证）
- **后续流程**：通过Automation编排后续步骤（发通知、创建工单等，允许部分失败重试）

## Action的四大类型

### 1. 创建操作（Create）

**业务场景**：新增一个对象

**示例1：创建新设备**
```
操作名称：添加新设备
必填字段：
  - 设备编号（String，唯一）
  - 设备类型（Enum：离心机 | 搅拌机 | 压缩机）
  - 所属车间（String）
  - 安装日期（Date）
选填字段：
  - 负责人（String）
  - 保修期至（Date）
  - 备注（String）
权限要求：
  - 角色：设备管理员
自动填充字段：
  - 创建时间 = 当前时间
  - 创建人 = 当前登录用户
  - 初始状态 = "待启用"
```

**示例2：创建采购订单**
```
操作名称：创建采购订单
必填字段：
  - 供应商（关联：供应商对象）
  - 采购物品清单（列表）
  - 预计交付日期（Date）
  - 总金额（Double）
选填字段：
  - 紧急程度（Enum）
  - 备注说明（String）
业务规则校验：
  - 总金额 > 10万时，必须填写"审批人"
  - 供应商信用等级必须 >= "良好"
权限要求：
  - 角色：采购专员或更高级别
```

---

### 2. 更新操作（Update）

**业务场景**：修改对象的某些字段

**示例1：更新设备状态**
```
操作名称：变更设备状态
可修改字段：
  - 设备状态（Enum：运行中 | 停机 | 维修中 | 已报废）
  - 变更原因（String，必填）
前置条件：
  - 当前状态不能是"已报废"（已报废设备不可变更）
权限要求：
  - 运维人员：可变更为"停机"或"维修中"
  - 设备经理：可变更为任何状态
自动记录：
  - 变更时间 = 当前时间
  - 变更人 = 当前用户
  - 变更历史 = 追加到状态变更日志
```

**示例2：修改供应商信息**
```
操作名称：更新供应商资料
可修改字段：
  - 联系电话
  - 公司地址
  - 业务联系人
不可修改字段：
  - 供应商编号（固定）
  - 注册时间（历史数据）
权限要求：
  - 采购专员：可修改联系信息
  - 采购经理：可修改所有字段（除固定字段外）
```

### 3. 删除操作（Delete）

**业务场景**：删除或停用对象

**注意**：真正的物理删除较少使用，更常见的是"软删除"（标记为"已作废"、"已归档"）。

**示例1：报废设备**
```
操作名称：申请设备报废
必填字段：
  - 报废原因（Enum：老化 | 损坏 | 淘汰 | 其他）
  - 详细说明（String）
前置条件：
  - 设备状态必须是"停机"
  - 设备使用年限 >= 5年，或累计故障次数 >= 10次
执行效果：
  - 设备状态 → "已报废"
  - 可用性 → false
  - 报废日期 → 当前日期
  - 触发后续流程 → 创建资产处置工单
权限要求：
  - 提交申请：设备管理员
  - 审批通过：设备经理（需二次确认）
```

**示例2：归档订单**
```
操作名称：归档历史订单
前置条件：
  - 订单状态必须是"已完成"或"已取消"
  - 订单完成日期 >= 1年前
执行效果：
  - 订单状态 → "已归档"
  - 在主列表中隐藏（但仍可查询）
权限要求：
  - 系统管理员或财务主管
```

### 4. 特殊业务操作（Custom Business Logic）

**业务场景**：除了增删改查以外的复杂业务操作

**示例1：紧急停机**
```
操作名称：紧急停机
前置条件：
  - 设备状态必须是"运行中"
必填字段：
  - 停机原因（Enum：设备故障 | 安全隐患 | 计划维护 | 其他）
执行效果：
  1. 发送停机指令到设备控制系统（硬件断电）
  2. 设备状态 → "停机"
  3. 记录停机时间
  4. 自动创建"故障报告"工单（如果原因是"设备故障"）
  5. 发送通知到企业微信群："【紧急】XX设备已停机"
权限要求：
  - 高级技术员及以上
  - 仅限工作时间（8:00-20:00），非工作时间需额外审批
安全确认：
  - 执行前弹窗二次确认："确定紧急停机？此操作将立即断电。"
```

**示例2：批量调价**
```
操作名称：批量调整产品价格
必填字段：
  - 产品类别（Enum）
  - 调价幅度（Double，可正可负）
  - 调价原因（String）
  - 生效日期（Date）
前置条件：
  - 调价幅度绝对值不能超过20%
业务逻辑：
  1. 查询所有该类别的产品
  2. 逐个计算新价格 = 旧价格 × (1 + 调价幅度)
  3. 批量更新产品价格
  4. 记录调价历史
  5. 生成调价通知文档（PDF）
权限要求：
  - 产品经理：可调价 ±10%
  - 总经理：可调价 ±20%
回滚机制：
  - 提供"撤销调价"操作（仅限24小时内）
```

**示例3：预约保养**
```
操作名称：预约设备保养
必填字段：
  - 保养类型（Enum：日常保养 | 深度保养 | 年检）
  - 预约日期（Date）
  - 预约时段（Enum：上午 | 下午 | 晚上）
前置条件：
  - 设备状态不能是"已报废"
  - 预约日期必须 >= 当前日期 + 3天（至少提前3天预约）
业务逻辑：
  1. 检查该时段是否已有其他设备保养预约
  2. 创建保养工单
  3. 自动分配保养工程师（根据工作负载）
  4. 发送提醒给工程师和设备负责人
  5. 在日历上标记保养时段
权限要求：
  - 设备负责人或运维人员
```

## Action设计技巧

### 技巧1：明确前置条件
在用户点击按钮前，就应该知道能否执行：
- ✅ 好的设计：当设备状态是"已报废"时，"启动设备"按钮为灰色（不可点击）
- ❌ 不好的设计：用户点击后才提示"该设备已报废，无法启动"

**进阶**：前置条件可以调用Function进行复杂校验（详见 `function-guide.md` 的"Function作为Action校验器"部分）。例如：
- 简单条件：`设备状态 ≠ "已报废"` → 直接写在Action中
- 复杂条件：`设备温度风险等级 ≠ "高风险"` → 调用Function判断

### 技巧2：提供必要的二次确认
对于高风险操作，增加确认步骤：
```
高风险操作（需二次确认）：
  - 删除、报废
  - 紧急停机、强制下线
  - 批量操作（影响 >10 个对象）
  - 不可逆操作（如发布到生产环境）
```

### 技巧3：自动填充能自动填充的字段
减少用户输入负担：
- 自动填充：创建时间、创建人、当前日期
- 智能推荐：根据历史数据推荐默认值
- 关联数据：选择供应商后，自动带出联系人和地址

### 技巧4：提供清晰的执行反馈
```
操作执行前 → 显示加载动画："正在停机..."
操作执行后 → 显示结果：
  ✅ 成功："设备已成功停机"
  ❌ 失败："停机失败：设备通信超时，请重试或联系技术支持"
```

### 技巧5：考虑回滚或撤销机制
对于重要操作，提供"后悔药"：
```
支持撤销的操作示例：
  - 删除 → 移到回收站（30天内可恢复）
  - 批量调价 → 24小时内可撤销
  - 发布公告 → 发布后1小时内可撤回
```

## 权限控制模式

### 1. 基于角色（Role-Based）
```
操作：审批采购订单
权限规则：
  - 采购专员：可查看，不可审批
  - 采购经理：可审批 ≤ 50万的订单
  - 财务总监：可审批任何金额
```

### 2. 基于条件（Conditional）
```
操作：修改客户信用额度
权限规则：
  - 如果 信用额度调整 ≤ 10%：客户经理可执行
  - 如果 信用额度调整 > 10%：需财务经理审批
  - 如果 客户有逾期记录：任何调整都需总经理批准
```

### 3. 基于时间（Time-Based）
```
操作：访问生产系统
权限规则：
  - 工作日 8:00-18:00：运维人员可直接访问
  - 其他时间：需提交申请并获得值班经理批准
```

### 4. 基于对象状态（State-Based）
```
操作：修改订单
权限规则：
  - 订单状态 = "草稿"：创建人可修改
  - 订单状态 = "待审批"：审批人可修改
  - 订单状态 = "已发货"：任何人不可修改
```

## 常见问题

### Q1: 什么时候应该创建一个新的Action？
**A**: 当用户需要"做一件事"时，就需要一个Action。例如：
- 用户需要"提交请假申请" → 创建Action
- 用户需要"查看设备列表" → 不需要Action（这是查询，不是操作）

### Q2: Action和Function的区别？
**A**:
- **Function**：只读，计算结果但不改数据（如"计算风险等级"）
- **Action**：可写，执行操作并修改数据（如"紧急停机"）

### Q3: 一个对象应该有多少个Action？
**A**: 通常3-10个就足够：
- 基础操作（必须）：创建、编辑、删除/停用
- 业务操作（按需）：2-7个特定业务场景的操作

### Q4: 如果操作很复杂，需要很多步骤怎么办？
**A**: 考虑使用"向导式表单"或拆分成多个小操作：
- ❌ 一个Action完成10个步骤 → 用户容易出错
- ✅ 拆分成3个Action，每个3-4个步骤 → 清晰可控

### Q5: Action失败了怎么办？
**A**: Action应该有完整的错误处理：
```
try:
  执行操作
  如果成功 → 提交事务，显示成功消息
except 业务规则校验失败:
  回滚事务，提示用户修正输入
except 系统错误:
  回滚事务，记录错误日志，提示"操作失败，请联系管理员"
```

## Action命名建议

**推荐做法** ✅：
- 使用动词开头：
  - "创建供应商"
  - "提交审批"
  - "启动设备"
  - "导出报表"
- 明确具体：
  - ✅ "紧急停机"
  - ❌ "操作设备"

**避免的做法** ❌：
- 过于技术：`update_device_status_v2()`
- 过于简略：`提交` `处理` `执行`
- 含糊不清：`操作1` `动作A`

## Action与Automation的配合

很多Action执行后，会触发Automation（自动化规则）：

```
用户点击"紧急停机" Action
   ↓
设备状态更新为"停机"
   ↓
Automation检测到状态变化
   ↓
自动执行后续操作：
  - 发送通知
  - 创建故障工单
  - 记录停机日志
```

设计Action时，考虑它会触发哪些自动化流程！
