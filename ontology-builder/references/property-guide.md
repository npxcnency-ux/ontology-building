# 数据字段类型指南（Property Guide）

本文档帮助你理解不同类型的数据字段。

## 基本字段类型

### 1. 文本（String）
**适用场景**：名称、描述、编号、地址等
**示例**：
- 设备编号：EQ-001
- 供应商名称：上海XX机械有限公司
- 产品描述：工业级离心机

### 2. 数字（Integer / Double）
**适用场景**：数量、金额、评分、测量值
**示例**：
- 库存数量：150（整数）
- 设备温度：87.5（小数）
- 供应商评分：4.5（小数）

### 3. 日期时间（Date / Timestamp）
**适用场景**：时间点、期限、记录时间
**示例**：
- 安装日期：2024-03-15
- 最后维护时间：2024-03-18 14:30:00
- 合同到期日：2025-12-31

### 4. 布尔值（Boolean）
**适用场景**：是/否、开/关、激活/停用
**示例**：
- 是否在保修期：是
- 设备是否在线：否
- 供应商是否认证：是

### 5. 枚举（Enum）
**适用场景**：固定选项的状态、分类
**示例**：
- 设备状态：运行中 | 停机 | 维修中 | 已报废
- 订单状态：待处理 | 处理中 | 已发货 | 已完成 | 已取消
- 优先级：低 | 中 | 高 | 紧急

### 6. 地理位置（GeoPoint）
**适用场景**：位置、坐标
**示例**：
- 设备位置：经度 121.5, 纬度 31.2
- 仓库地址：上海市浦东新区XX路XX号

## 时序属性（Time Series Property）

某些字段的值会随时间变化，你可能需要查看历史趋势。

### 何时使用时序属性？
如果你需要回答以下问题，就应该使用时序属性：
- ❓ 过去24小时设备温度如何变化？
- ❓ 上个月库存量的波动趋势？
- ❓ 这个客户的信用评分历史记录？

### 时序属性示例

**设备温度（时序）**
- 当前值：87.5°C
- 历史查询：
  - 2024-03-18 14:00 → 85.2°C
  - 2024-03-18 14:15 → 86.8°C
  - 2024-03-18 14:30 → 87.5°C

**非时序 vs 时序对比**

| 字段 | 是否时序 | 原因 |
|------|---------|------|
| 设备编号 | ❌ 否 | 固定不变 |
| 设备名称 | ❌ 否 | 很少变化，即使变化也不需要历史记录 |
| 实时温度 | ✅ 是 | 需要查看温度趋势图 |
| 设备状态 | ✅ 是 | 需要追溯"什么时候停机的" |
| 累计运行时长 | ✅ 是 | 需要分析运行时长增长曲线 |
| 所属车间 | ❌ 否 | 偶尔变化，但不需要历史趋势 |

## 常见问题

### Q1: 我不确定某个字段应该用什么类型？
**A**: 先用最简单的类型。例如：
- 如果只是记录文字 → 用文本（String）
- 如果需要计算 → 用数字（Integer/Double）
- 如果是固定几个选项 → 用枚举（Enum）

### Q2: 什么时候用整数（Integer），什么时候用小数（Double）？
**A**:
- 整数：数量、人数、次数（如库存150件、员工25人）
- 小数：测量值、评分、百分比（如温度87.5°C、评分4.5分）

### Q3: 时序属性会增加成本吗？
**A**: 会占用更多存储空间，但对于关键监控指标（如设备温度、库存量）非常值得。建议只对真正需要查看历史趋势的字段启用时序。

### Q4: 字段太多了，怎么办？
**A**: 优先保留核心字段，次要字段可以后续补充。一般建议：
- 核心对象：5-15个字段
- 复杂对象：15-30个字段
- 如果超过30个字段，考虑拆分成多个关联对象

## 字段命名建议

**推荐做法** ✅：
- 使用清晰的业务名称：`设备温度` 而非 `temp`
- 包含单位：`温度(°C)` `重量(kg)` `价格(元)`
- 保持一致性：如果用"编号"，就不要混用"ID"

**避免的做法** ❌：
- 过于技术化：`device_temp_sensor_value_01`
- 过于简写：`qty` `amt` `desc`
- 含糊不清：`数据1` `字段A` `信息`
