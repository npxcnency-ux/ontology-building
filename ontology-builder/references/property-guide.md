# 数据字段类型指南（Property Guide）

本文档帮助你理解不同类型的数据字段。

## 基本字段类型

### 1. 文本（String）
**适用场景**：名称、描述、编号、地址等
**示例**：
- 设备编号：EQ-001
- 供应商名称：上海XX机械有限公司
- 产品描述：工业级离心机

### 2. 数字（Integer / Double）
**适用场景**：数量、金额、评分、测量值
**示例**：
- 库存数量：150（整数）
- 设备温度：87.5（小数）
- 供应商评分：4.5（小数）

### 3. 日期时间（Date / Timestamp）
**适用场景**：时间点、期限、记录时间
**示例**：
- 安装日期：2024-03-15
- 最后维护时间：2024-03-18 14:30:00
- 合同到期日：2025-12-31

### 4. 布尔值（Boolean）
**适用场景**：是/否、开/关、激活/停用
**示例**：
- 是否在保修期：是
- 设备是否在线：否
- 供应商是否认证：是

### 5. 枚举（Enum）
**适用场景**：固定选项的状态、分类
**示例**：
- 设备状态：运行中 | 停机 | 维修中 | 已报废
- 订单状态：待处理 | 处理中 | 已发货 | 已完成 | 已取消
- 优先级：低 | 中 | 高 | 紧急

#### 状态字段的生命周期设计

当枚举字段用于表示对象"状态"时，需要设计**状态流转规则**——哪些状态之间可以互相转换。

**示例：设备状态的生命周期**
```
待启用 → 运行中 → 停机 → 运行中（可恢复）
                  ↓
               维修中 → 运行中（修复后恢复）
                  ↓
               已报废（终态，不可逆）

合法流转：
  ✅ 待启用 → 运行中（首次启动）
  ✅ 运行中 → 停机（正常停机或紧急停机）
  ✅ 停机 → 运行中（恢复运行）
  ✅ 停机 → 维修中（需要维修）
  ✅ 维修中 → 运行中（修复完成）
  ✅ 停机 → 已报废（申请报废）

非法流转：
  ❌ 已报废 → 运行中（报废不可逆）
  ❌ 待启用 → 已报废（未使用不能报废）
  ❌ 运行中 → 已报废（必须先停机）
```

**为什么要设计状态流转？**
- 状态流转 = Action的前置条件（"启动设备"要求当前状态="停机"）
- 非法流转 = 业务规则约束（防止误操作）
- 终态识别 = 哪些状态是不可逆的（如"已报废""已取消"）

**💡 设计建议**：
1. 每个状态对象至少考虑：**初始态**、**工作态**、**终态**
2. 明确哪些是**终态**（不可逆），如"已报废""已归档""已取消"
3. 状态数量建议3-7个，超过7个考虑是否可以合并

### 6. 地理位置（GeoPoint）
**适用场景**：位置、坐标
**示例**：
- 设备位置：经度 121.5, 纬度 31.2
- 仓库地址：上海市浦东新区XX路XX号

## 时序属性（Time Series Property）

某些字段的值会随时间变化，你可能需要查看历史趋势。

### 何时使用时序属性？
如果你需要回答以下问题，就应该使用时序属性：
- ❓ 过去24小时设备温度如何变化？
- ❓ 上个月库存量的波动趋势？
- ❓ 这个客户的信用评分历史记录？

### 时序属性示例

**设备温度（时序）**
- 当前值：87.5°C
- 历史查询：
  - 2024-03-18 14:00 → 85.2°C
  - 2024-03-18 14:15 → 86.8°C
  - 2024-03-18 14:30 → 87.5°C

**非时序 vs 时序对比**

| 字段 | 是否时序 | 原因 |
|------|---------|------|
| 设备编号 | ❌ 否 | 固定不变 |
| 设备名称 | ❌ 否 | 很少变化，即使变化也不需要历史记录 |
| 实时温度 | ✅ 是 | 需要查看温度趋势图 |
| 设备状态 | ✅ 是 | 需要追溯"什么时候停机的" |
| 累计运行时长 | ✅ 是 | 需要分析运行时长增长曲线 |
| 所属车间 | ❌ 否 | 偶尔变化，但不需要历史趋势 |

## 常见问题

### Q1: 我不确定某个字段应该用什么类型？
**A**: 先用最简单的类型。例如：
- 如果只是记录文字 → 用文本（String）
- 如果需要计算 → 用数字（Integer/Double）
- 如果是固定几个选项 → 用枚举（Enum）

### Q2: 什么时候用整数（Integer），什么时候用小数（Double）？
**A**:
- 整数：数量、人数、次数（如库存150件、员工25人）
- 小数：测量值、评分、百分比（如温度87.5°C、评分4.5分）

### Q3: 时序属性会增加成本吗？
**A**: 会占用更多存储空间，但对于关键监控指标（如设备温度、库存量）非常值得。建议只对真正需要查看历史趋势的字段启用时序。

### Q3b: 时序数据需要保留多久？
**A**: 不同类型的数据保留需求不同，需要在配置时明确保留策略：

| 数据类型 | 建议保留周期 | 理由 |
|---------|------------|------|
| 实时传感器数据（温度、压力） | 30-90天 | 趋势分析通常看近期 |
| 设备状态变更历史 | 1-2年 | 故障分析和审计需要 |
| 业务指标（利用率、评分） | 1年 | 年度对比分析 |
| 合规审计数据 | 按法规要求（3-7年） | 法律法规强制要求 |

**成本估算参考**：
```
1000台设备 × 每分钟采集 × 保留90天
= 1000 × 60 × 24 × 90 = 1.3亿条记录
→ 约占用 50-100GB 存储

如果保留1年：
= 1000 × 60 × 24 × 365 = 5.3亿条记录
→ 约占用 200-400GB 存储
```

**建议**：从短周期开始（如90天），根据实际使用再调整。过期数据可以归档到低成本存储，而非直接删除。

### Q4: 字段太多了，怎么办？
**A**: 优先保留核心字段，次要字段可以后续补充。一般建议：
- 核心对象：5-15个字段
- 复杂对象：15-30个字段
- 如果超过30个字段，考虑拆分成多个关联对象

### Q5: 哪些字段应该设为必填？
**A**: 用以下三条标准判断：

| 标准 | 必填 | 选填 | 示例 |
|------|------|------|------|
| 没有它对象就无法识别？ | ✅ | | 设备编号、工单号 |
| 业务流程必须依赖它？ | ✅ | | 设备类型（决定保养规则）、订单金额 |
| 创建时不一定有，后续补充？ | | ✅ | 负责人、备注、附件 |

**经验法则**：
- 主键字段 → 必须必填
- 枚举状态字段 → 通常必填（可设默认值如"待启用"）
- 描述/备注类字段 → 通常选填
- 外部系统同步的字段 → 选填（数据由系统自动填入，不需要用户手动填写）
- 建议必填字段 ≤ 7个（超过会增加用户填写负担）

## 字段命名建议

**推荐做法** ✅：
- 使用清晰的业务名称：`设备温度` 而非 `temp`
- 包含单位：`温度(°C)` `重量(kg)` `价格(元)`
- 保持一致性：如果用"编号"，就不要混用"ID"

**避免的做法** ❌：
- 过于技术化：`device_temp_sensor_value_01`
- 过于简写：`qty` `amt` `desc`
- 含糊不清：`数据1` `字段A` `信息`

## 数据来源映射

每个字段的数据从哪里来？这直接影响系统如何获取和维护数据。

### 三类数据来源

| 来源 | 说明 | 示例 |
|------|------|------|
| 人工录入 | 用户在界面填写 | 设备名称、负责人、备注 |
| 外部系统同步 | 从ERP、MES、传感器等自动导入 | 设备温度、库存数量、订单状态 |
| 系统自动生成 | 系统根据规则自动产生 | 创建时间、唯一编号、审计日志 |

### 外部系统同步：需要明确数据管道

当数据来自外部系统时，必须回答以下问题：

```
1. 数据从哪个系统来？
   → 例如：SAP、MES、IoT传感器平台

2. 具体来自哪张表/哪个接口？
   → 例如：SAP的EQUI表、传感器平台的/api/v1/readings

3. 对应哪个字段？
   → 例如：EQUI.HERST（制造商）、readings.temperature

4. 多久同步一次？
   → 实时推送、每5分钟、每小时、每天凌晨
```

**示例：设备对象的数据来源映射**

| 字段 | 来源 | 源系统 | 源表/接口 | 源字段 | 同步频率 |
|------|------|--------|----------|--------|---------|
| 设备编号 | 外部同步 | SAP | EQUI | EQUI.EQUNR | 每天 |
| 设备名称 | 外部同步 | SAP | EQUI | EQUI.EQKTX | 每天 |
| 实时温度 | 外部同步 | IoT平台 | /api/readings | temperature | 实时 |
| 设备状态 | 人工录入 | — | — | — | — |
| 负责人 | 人工录入 | — | — | — | — |
| 创建时间 | 系统生成 | — | — | — | — |

**💡 提示**：不确定源系统细节也没关系。先标注"来自XX系统"，具体的表和字段可以后续与IT团队确认。重要的是**明确数据不是凭空来的**。
