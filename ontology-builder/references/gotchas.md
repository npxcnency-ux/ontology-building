# 本体构建常见问题与陷阱 (Gotchas)

这份文档收集了在企业本体构建过程中最容易犯的错误和需要特别注意的地方。

---

## 🚨 设计阶段陷阱

### Gotcha 1: 一开始就想设计"完美"的本体
**问题**: 企业白领往往想一次性设计出包含所有功能的完整本体,导致项目复杂度爆炸,迟迟无法上线。

**为什么会出现**:
- 担心"遗漏重要功能"
- 想一步到位避免后续修改
- 不确定哪些功能是必需的

**正确做法** ✅:
- **MVP原则**: 先实现最核心的3-5个功能
- **迭代优化**: 上线后根据用户反馈逐步增加功能
- **80/20法则**: 先满足80%的常见场景,特殊情况后续再处理

**例子**:
```
❌ 第一版就设计:
   - 15个Property
   - 8个Function
   - 12个Action
   - 6个Automation
   结果: 6个月还没完成

✅ 第一版先做:
   - 5个核心Property (设备编号、状态、温度等)
   - 1个Function (温度风险判断)
   - 2个Action (创建设备、变更状态)
   - 1个Automation (高温自动停机)
   结果: 2周上线,3个月后迭代到完整版
```

**对象识别也一样——不要焦虑第一次列全**:
```
对象之间存在"引力"——建模过程本身会揭示缺失的对象。

实例（整车质量预告警，最终15个对象）：
  warningEvent → 拉出 workOrder（事件需要跟进）
  workOrder → 拉出 handlingRecord 和 cardPushRecord（工单需要执行和通知）
  effectivenessMetric → 拉出 iterationAnalysis（效果评估需要分析支撑）
  iterationAnalysis → 拉出 backtestTask 和 shadowRunTask（分析结论需要验证）

第一轮识别了 8 个对象，后续 7 个都是建模过程中被"拉出来"的。
```

阶段一先识别核心对象（通常 5-8 个），建模过程中发现需要新对象时回来补充，用"骨架→完整"的渐进式策略。

### Gotcha 2: 混淆Property和Function
**问题**: 不知道某个数据应该存为Property还是用Function计算。

**判断标准**:
| 特征 | Property (属性) | Function (计算规则) |
|------|----------------|---------------------|
| 存储方式 | 存在数据库 | 实时计算 |
| 谁来赋值 | 人工输入 或 外部系统写入 | 根据其他数据自动计算 |
| 典型例子 | 设备编号、安装日期 | 风险等级、在线状态 |

**常见错误**:
- ❌ 把"风险等级"设为Property,每次都要人工判断
  ✅ 应该用Function根据温度自动计算
- ❌ 把"设备名称"设为Function返回值
  ✅ 应该是Property,人工填写或导入

**记忆技巧**: 如果你能用"如果...那么..."描述这个数据,它应该是Function。
- "如果温度>90°C,那么风险等级=高" → Function ✅
- "设备编号就是EQ-001" → Property ✅

### Gotcha 3: Function设计得过于复杂
**问题**: Function的计算逻辑太复杂,包含多层嵌套判断、循环、外部API调用。

**为什么是问题**:
- Function会被频繁调用(界面显示、Automation判断),复杂逻辑会拖慢系统
- 复杂逻辑难以维护和调试
- 可能引入性能瓶颈

**建议**:
```
Function的计算逻辑应该:
✅ 在 < 100行代码以内
✅ 只依赖对象自身的Property
✅ 不调用外部API (除非性能可控)
✅ 避免复杂查询或大数据量遍历

如果计算逻辑很复杂:
→ 考虑是否应该拆分成多个Function
→ 考虑是否应该改用Automation (后台异步执行)
→ 考虑是否应该预先计算并存为Property
```

### Gotcha 4: 忘记设置Action的前置条件
**问题**: 用户可以在任何状态下执行任何操作,导致数据混乱。

**常见错误场景**:
```
❌ 已报废的设备还能"启动"
❌ 已完成的订单还能"取消"
❌ 设备温度正常时也能"紧急停机"
```

**正确做法** ✅:
```
每个Action都应该明确前置条件:

Action: 启动设备
前置条件:
  - 设备状态 = "停机" (不能是"已报废")
  - 设备是否在线 = "在线"
  - 设备温度 < 温度标准

Action: 取消订单
前置条件:
  - 订单状态 = "待处理" 或 "处理中"
  - 不能是"已发货" 或 "已完成"
```

### Gotcha 5: Automation的触发频率与限流保护
**问题**: Automation触发太频繁（淹没通知渠道）或从不触发（名称写错）。

**触发太频繁**:
```
❌ 触发条件: 温度 > 80°C
结果: 温度在80-90°C之间正常波动,每次波动都触发,一天触发100次

✅ 改进: 温度 > 90°C 且持续 > 5分钟
结果: 只有真正异常时才触发

❌ 库存不足提醒:
   每次库存减少1件都发通知 → 一天1000条消息
   用户直接屏蔽所有通知

✅ 改进:
   - 同一商品24小时内最多提醒1次
   - 或者只在每天早上汇总一次性提醒
```

**从不触发**:
```
❌ 触发条件: Function "设备风险等级" == "高风险"
问题: Function名称写错了,实际是"设备温度风险等级"
结果: 条件永远不满足,Automation从不触发

✅ 测试方法:
1. 先手动构造触发条件 (如设置温度为100°C)
2. 观察Automation是否真的执行了
3. 检查日志确认触发原因
```

**限流配置建议**:
```
对于异常响应类Automation:
  - 同一对象N分钟内最多触发1次
  - 或设置冷却期: 触发后X分钟内不再触发

对于定时任务类Automation:
  - 合理设置执行频率 (不要设太频繁)
  - 批量处理而非逐个处理

对于流程编排类Automation:
  - 检测重复执行 (避免循环触发)
```

## ⚠️ 数据建模陷阱

### Gotcha 6: Property字段类型选错
**常见错误**:
| 错误做法 | 正确做法 | 影响 |
|---------|---------|------|
| 温度用String存"87.5°C" | 用Double存87.5 | 无法做数值比较和计算 |
| 日期用String存"2024-03-18" | 用Date类型 | 无法做日期范围查询 |
| 状态用String存任意文本 | 用Enum限定选项 | 用户可能输入错误值 |
| 编号用Integer存123 | 用String存"EQ-123" | 无法处理带前缀的编号 |

**记忆技巧**:
- 需要计算 → 数字类型 (Integer/Double)
- 需要日期比较 → Date/Timestamp
- 只有固定几个选项 → Enum
- 其他 → String

---

### Gotcha 7: 过度使用时序属性
**问题**: 把所有Property都设为时序属性,导致存储成本暴涨。

**判断标准**:
```
只对以下情况使用时序属性:
✅ 需要查看趋势图 (如温度变化曲线)
✅ 需要追溯历史值 (如"3天前库存是多少")
✅ 数据更新非常频繁 (如传感器数据)

不需要时序的情况:
❌ 设备编号 (固定不变)
❌ 设备名称 (很少变化)
❌ 所属车间 (偶尔变化,但不关心历史)
```

**成本对比**:
```
假设有1000台设备,每台每分钟上报一次温度:

非时序属性:
  - 存储量: 1000条记录 (每台设备只存当前值)

时序属性:
  - 存储量: 1000 × 60 × 24 × 30 = 4320万条记录/月
  - 存储成本高100倍以上
```

---

### Gotcha 8: 字段命名不清晰
**问题**: 使用技术化或含糊的字段名,业务人员看不懂。

**常见错误**:
```
❌ 不清晰的命名:
   - temp (什么的温度?)
   - status (哪个状态?)
   - flag1, flag2 (什么意思?)
   - data, value (太泛化)

✅ 清晰的命名:
   - 设备温度(°C)
   - 设备运行状态
   - 是否需要保养
   - 实时库存数量
```

**命名原则**:
- 使用完整的业务术语
- 包含单位 (温度(°C)、重量(kg))
- 避免缩写 (除非是业内通用的)
- 保持一致性 (统一用"编号"或"ID",不要混用)

## 🔧 实施阶段陷阱

### Gotcha 9: 没有充分测试就上线
**问题**: 直接在生产环境部署,出现问题后影响业务。

**必须测试的场景**:
```
✅ 正常场景:
   - 创建对象 → 成功
   - 执行操作 → 正常
   - 自动化触发 → 如预期

✅ 异常场景:
   - 输入不合法数据 → 是否正确拦截?
   - 不满足前置条件执行Action → 是否报错?
   - Function计算遇到空值 → 是否崩溃?

✅ 边界场景:
   - 温度刚好等于90°C → 是高风险还是低风险?
   - 同时有100个用户创建设备 → 系统是否正常?
   - Automation执行失败 → 是否有错误处理?
```

**测试环境建议**:
1. 先在测试环境部署
2. 用真实或接近真实的数据测试
3. 邀请3-5个实际用户试用1周
4. 收集反馈并优化
5. 才正式上线生产环境

---

### Gotcha 10: 权限控制设置不当
**问题**: 给错误的角色授予了高危操作权限,或限制太严导致正常操作无法执行。

**常见错误**:
```
❌ 过于宽松:
   - 普通员工可以"删除所有数据"
   - 实习生可以"审批百万级采购"

❌ 过于严格:
   - 运维人员无法"变更设备状态"
   - 采购专员无法"创建采购订单"
```

**权限设计原则**:
```
✅ 最小权限原则:
   - 只给用户完成工作所需的最小权限
   - 高危操作 (删除、大额审批) 需要高权限

✅ 职责分离:
   - 申请和审批不能是同一人
   - 操作和验证应该分开

✅ 基于业务规则:
   - 采购专员可创建 ≤ 10万的订单
   - 采购经理可创建 ≤ 100万的订单
   - 超过100万需要总经理审批
```

---

### Gotcha 11: 忘记处理异常情况
**问题**: 只考虑了"一切顺利"的情况,没有处理失败场景。

**必须考虑的异常**:
```
Action执行失败:
  ✅ 设备通信超时 → 重试3次后提示用户
  ✅ 审批人不存在 → 提示"无效的审批人"
  ✅ 数据校验失败 → 明确提示哪个字段有问题

Function计算异常:
  ✅ 温度数据为空 → 返回"数据不足,无法判断"
  ✅ 除数为0 → 返回"计算错误"

Automation执行失败:
  ✅ 发送通知失败 → 记录日志,稍后重试
  ✅ 调用Action失败 → 通知管理员人工处理
```

## 💡 用户体验陷阱

### Gotcha 12: 操作按钮的用户体验——点击前清晰、点击后有反馈
**问题**: 用户不知道点击按钮会发生什么,点击后也不知道是否成功。

**点击前——按钮文案要清晰**:
```
❌ 按钮文字:
   - "提交" (提交什么?)
   - "确定" (确定什么?)
   - "处理" (怎么处理?)

✅ 清晰的文字:
   - "提交报废申请"
   - "确定紧急停机"
   - "批准采购订单"
```

**确认弹窗要有足够信息**:
```
❌ "确定删除吗?"
✅ "确定删除设备'1#离心机'吗?删除后无法恢复。"
```

**点击后——必须有明确反馈**:
```
✅ 操作执行中:
   - 显示加载动画 + 提示文字: "正在停机,请稍候..."

✅ 操作成功:
   - 成功提示: "设备已成功停机"
   - 显示接下来可以做什么: "查看停机日志" "创建维修工单"

✅ 操作失败:
   - 失败提示: "停机失败: 设备通信超时"
   - 告知用户可以怎么做: "请检查设备网络连接,或联系技术支持"

✅ 操作需要等待:
   - 提示: "报废申请已提交,等待设备经理审批"
   - 提供跟踪链接: "查看审批进度"
```

### Gotcha 13: 表单字段太多,用户填写负担重
**问题**: 创建对象时需要填写20+个字段,用户体验极差。

**优化方法**:
```
✅ 分清必填和选填:
   - 核心字段 (3-5个) → 必填
   - 其他字段 → 选填,可以后续补充

✅ 智能默认值:
   - 创建时间 → 自动填充当前时间
   - 创建人 → 自动填充当前用户
   - 状态 → 默认为"草稿"

✅ 关联数据自动带出:
   - 选择供应商后,自动填充联系人和地址
   - 选择产品后,自动填充价格和库存

✅ 分步表单:
   - 第一步: 基本信息 (3个字段)
   - 第二步: 详细信息 (5个字段)
   - 第三步: 附件和备注 (选填)
```

## 🛠️ 维护阶段陷阱

### Gotcha 14: 配置变更管理——记录历史、小步迭代
**问题**: 修改了配置但没有记录,或者一次性改太多,出问题后不知道是哪里导致的。

**变更历史**:
```
✅ 版本控制:
   - 每次修改配置文件后,标注版本号
   - 记录变更内容和原因
   - 保留历史版本以便回滚

✅ 变更日志示例:
   Version 1.1.0 (2024-03-18)
   - 修改: 温度风险判断从 > 85°C 改为 > 90°C
   - 原因: 用户反馈误报太多
   - 影响: Automation触发频率降低约50%

   Version 1.0.0 (2024-03-01)
   - 初始版本
```

**小步迭代**:
```
✅ 每次只修改1-2个配置项
   测试验证无误后再修改下一个
   这样出问题容易定位

❌ 一次性改10个配置
   上线后发现不工作
   不知道是哪个改错了
```

## 📝 文档陷阱

### Gotcha 15: 没有给业务人员提供操作手册
**问题**: 技术配置写得很完整,但业务人员不知道怎么用。

**必须提供的文档**:
```
✅ 用户操作手册:
   - 如何创建设备
   - 如何查看设备状态
   - 如何申请报废

✅ 业务流程说明:
   - 设备报废审批流程图

✅ 培训材料:
   - 快速上手视频 (5分钟)
   - 操作截图指南
   - 典型场景演示
```

### Gotcha 16: 配置文件和文档不一致
**问题**: 修改了配置但忘记更新文档,用户按文档操作却不工作。

**建议**:
```
✅ 同步更新:
   - 修改ontology-config.yaml时
   - 同时更新ontology-documentation.md
   - 检查用户手册是否需要更新

✅ 定期审查:
   - 每季度检查一次文档与实际配置是否一致
   - 修正发现的不一致
```

### Gotcha 17: Link和Property混淆——该建关系的存成了文本
**问题**: 把应该用Link关联的对象，存成了一个文本Property。

**常见错误场景**:
```
❌ "所属车间" 用String存"A栋2楼"
   → 如果"车间"有自己的属性（面积、负责人、设备列表），它就是独立对象
   ✅ 应该用Link关联到独立的"车间"对象

❌ "负责技师" 用String存"张三"
   → 如果"技师"有自己的属性（工号、技能证书、排班表），它就是独立对象
   ✅ 应该用Link关联到独立的"技师"对象
```

**判断标准**:
- 如果关联目标是一个**需要独立管理的对象**（有自己的属性和操作）→ 用Link
- 如果只是记录一个**简单的文本值**（不需要独立管理）→ 用Property

**另一个常见错误——过度关联**:
```
❌ 设备 → 财务报表（没有直接业务关系）
✅ 设备 → 工单 → 费用明细 → 财务报表（通过中间对象间接关联）
```

## ✅ 最佳实践总结

### 设计阶段
1. 先做MVP,再迭代（不要追求第一版完美,对象会在建模过程中被"拉出来"）
2. Property vs Function: 能自动计算的用Function
3. Function保持简单 (< 100行代码)
4. 每个Action必须有前置条件
5. Automation要有限流保护,触发条件要可测试

### 数据建模
6. 字段类型选择: 需要计算→数字,固定选项→Enum
7. 谨慎使用时序属性 (只对需要趋势分析的字段使用)
8. 字段命名清晰 (用业务语言,包含单位)

### 实施阶段
9. 充分测试 (正常、异常、边界场景)
10. 权限控制要平衡 (不能太宽松也不能太严格)
11. 必须处理异常情况 (不要只考虑"一切顺利")

### 用户体验
12. 操作按钮：点击前文案清晰,点击后反馈明确
13. 表单字段精简 (3-5个必填,其他选填)

### 维护阶段
14. 配置变更管理：记录历史 + 小步迭代
15. 提供业务人员操作手册
16. 保持配置和文档一致

---

记住: **先把核心功能做好,再慢慢扩展。完成的70分方案,胜过永远在设计中的100分方案!**

## 🧠 大规模建模心得（多对象场景）

以下来自一次完整的15对象建模实战（整车质量预告警系统），适用于对象数量≥5的复杂业务场景。

---

### Gotcha 18: 把该独立的对象塞成了字段——用"操作独立性"判断边界

**问题**: 不确定某个概念应该是独立对象还是几个字段。

**判断标准**:
```
如果某个概念需要独立地被 创建、修改、查询、删除，它就是独立对象。

❌ 标注信息 做成 warningEvent 的字段（annotationLabel, annotator, annotationTime）
   → 丧失了"一个事件被多次标注、标注有独立生命周期"的能力

✅ 标注信息 拆为独立的 annotationRecord 对象
   → 支持多次标注、独立查询、独立权限控制
```

**记忆技巧**: 不是由数据结构决定的，而是由**业务操作的独立性**决定的。如果你能对它单独执行 CRUD，它就是对象。

---

### Gotcha 19: 文本字段包装了本该是结构化关联的数据——重构信号

**问题**: 早期用文本字段（如JSON字符串、逗号分隔列表）存储了本该用Link关联的数据，随着建模推进，暴露出无法复用、无法校验、无法关联查询的问题。

**重构时机信号**:
```
当你发现需要在两个地方维护同一份信息时，就该重构了。

❌ failureCause 里有 signalNames（文本列表），同时 signalDefinition 作为独立对象也存在
   → 同一份信号信息存了两处，修改时必然不同步

✅ 删掉 signalNames 文本字段，改用 M:N Link 关联到 atomicCondition
   → 单一数据源，修改自动同步
```

**操作建议**: 重构时机通常在建到第 8-12 个对象时最明显——此时对象间的复用需求开始涌现。不要在第 3 个对象时强行预判，也不要在第 15 个对象时才补救。

---

### Gotcha 20: 过早抽象——抽象父类是"后验"的，不是"先验"的

**问题**: 在只有1-2个相似对象时就急于提取抽象父类，结果抽错了。

**正确做法**:
```
❌ 建完 backtestTask 后立即抽象出 AbstractVerificationTask
   → 此时只有1个实例，你不知道后面的 shadowRunTask 是否真的共享同样的字段结构

✅ 等 backtestTask 和 shadowRunTask 都建完后，发现它们字段高度相似，再提取抽象父类
   → 有至少2-3个具体实例佐证，抽象才可靠
```

**规则**: 好的抽象需要至少3个具体实例才能提取，2个可能是巧合。过早抽象是建模的大忌。

---

### Gotcha 21: M:N关系是复杂度的分水岭——不要轻易引入，但该用时别犹豫

**问题**: 不确定两个对象之间是1:N还是M:N关系。

**判断标准**:
```
选择 M:N 的标准不是"现在是否需要"，而是"复用是否有业务价值"。

案例：atomicCondition ↔ failureCause
  - "发动机水温超过105°C持续30秒" 这个原子工况
  - 既可能是"冷却系统失效"的触发条件之一
  - 也可能是"散热器堵塞"的触发条件之一
  → 复用有明确的业务价值 → M:N ✅

案例：workOrder ↔ warningEvent
  - 一个工单通常只对应一个预警事件
  - 不存在"同一个工单服务于多个事件"的业务场景
  → 1:N 足够 ✅
```

**注意**: M:N 意味着需要中间表（junction table），关系本身可能携带属性（优先级、排序、生效时间）。引入前确认这个复杂度是值得的。

---

### Gotcha 22: 跨对象引用一致性——大规模建模最难的不是设计，是命名

**问题**: 对象数量超过10个后，交叉引用检查（P0-6/7/9）手工核对极易出错。最难发现的问题不是"引用了不存在的东西"，而是"引用了名字很像但语义不同的东西"。

**常见错误**:
```
❌ 自动化规则里写 callAction: closeWorkOrder
   实际定义的操作按钮叫 completeWorkOrder
   → close（关闭） ≠ complete（完成），语义不同但快速扫描时容易滑过

❌ 前置条件引用 activePurchaseOrderCount
   实际定义的字段叫 pendingOrderCount
   → 两个名字都"听起来对"，但指向不同的业务含义
```

**防范措施**:
- 命名规范不是美学问题，是**正确性问题**
- 每新增一个对象后，立即更新 apiName 清单（Property / Action / Function 各一张）
- 写引用时从清单中复制粘贴，不要凭记忆手打
- 对象数量 > 10 时，建议用工具辅助做交叉引用校验

### Gotcha 23: 闭环设计模式——监控/预警类场景的通用架构

**问题**: 监控、预警、质量管理类场景中，对象数量多且关系复杂，不知道如何组织。

**通用模式**:
```
感知层 → 执行层 → 评估层 → 迭代验证层 → 回到感知层
  ↑                                           |
  └───────────────── 闭环 ──────────────────────┘

感知层：定义规则 + 检测异常
  配置对象（故障树、筛选模型、信号定义、原子工况）
  事件对象（预告警事件）

执行层：采取行动
  工单、推卡记录、处置记录、标注记录

评估层：评估效果
  效果指标、迭代分析记录

迭代验证层：优化规则
  回测任务、影子运行任务
  → 验证结果反馈到感知层的配置对象，形成闭环
```

**为什么有用**:
- 在阶段一就建立全局视角，知道对象属于哪一层
- 层内对象关系紧密，层间对象通过明确接口连接
- 识别闭环连接点（如 faultTreeConfig 同时出现在感知层和迭代验证层）

**适用场景**: 设备监控、质量预警、安全巡检、合规审计——所有"检测→响应→评估→优化"的业务场景。

---

### Gotcha 24: 区分"配置对象"和"记录对象"——它们的权限和审计需求完全不同

**问题**: 所有对象的权限和审计策略一刀切，导致配置对象被随意修改（引发连锁问题）或记录对象审批过重（拖慢业务效率）。

**两类对象的区别**:
| 特征 | 配置对象 | 记录对象 |
|------|---------|---------|
| 典型代表 | 故障树配置、筛选模型、信号定义 | 预警事件、工单、处置记录 |
| 变化频率 | 低（周/月级别） | 高（分钟/小时级别） |
| 被引用方式 | 被大量记录对象引用 | 引用配置对象 |
| 修改影响 | 牵一发动全身 | 只影响单条记录 |

**设计建议**:
```
配置对象：
  ✅ 修改权限应更高（管理员/专家级别）
  ✅ 需要变更历史和审计日志
  ✅ 关键修改需要审批流程
  ✅ 建议支持版本管理（如 faultTreeVersion）

记录对象：
  ✅ 创建权限可以更宽（一线操作人员）
  ✅ 审计需求较低（标准操作日志即可）
  ✅ 流程要轻量，不要过度审批
```

**常见错误**:
```
❌ 普通运维人员可以直接修改故障树配置
   → 一次误操作可能导致所有预警规则失效

❌ 创建一个工单需要三级审批
   → 预警事件响应被审批流程拖慢，失去时效性
```

**识别方法**: 如果一个对象有 `version` 或 `status`（draft/published）字段，它大概率是配置对象。

## ✅ 大规模建模最佳实践补充

### 多对象场景（≥5个对象）
18. 用"操作独立性"判断对象边界（能独立CRUD→独立对象）
19. 文本字段包装结构化数据是重构信号（两处维护同一信息→该用Link）
20. 抽象父类需要≥3个具体实例佐证（2个可能是巧合）
21. M:N关系按"复用的业务价值"判断（不是"现在是否需要"）
22. 对象 > 10个时，命名一致性是最大风险（用清单+复制粘贴，不要凭记忆）
23. 监控/预警类场景用闭环模式组织（感知→执行→评估→迭代）
24. 配置对象和记录对象需要不同的权限与审计策略
