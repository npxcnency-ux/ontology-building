# 本体构建常见问题与陷阱 (Gotchas)

这份文档收集了在企业本体构建过程中最容易犯的错误和需要特别注意的地方。

---

## 🚨 设计阶段陷阱

### Gotcha 1: 一开始就想设计"完美"的本体
**问题**: 企业白领往往想一次性设计出包含所有功能的完整本体,导致项目复杂度爆炸,迟迟无法上线。

**为什么会出现**:
- 担心"遗漏重要功能"
- 想一步到位避免后续修改
- 不确定哪些功能是必需的

**正确做法** ✅:
- **MVP原则**: 先实现最核心的3-5个功能
- **迭代优化**: 上线后根据用户反馈逐步增加功能
- **80/20法则**: 先满足80%的常见场景,特殊情况后续再处理

**例子**:
```
❌ 第一版就设计:
   - 15个Property
   - 8个Function
   - 12个Action
   - 6个Automation
   结果: 6个月还没完成

✅ 第一版先做:
   - 5个核心Property (设备编号、状态、温度等)
   - 1个Function (温度风险判断)
   - 2个Action (创建设备、变更状态)
   - 1个Automation (高温自动停机)
   结果: 2周上线,3个月后迭代到完整版
```

---

### Gotcha 2: 混淆Property和Function
**问题**: 不知道某个数据应该存为Property还是用Function计算。

**判断标准**:
| 特征 | Property (属性) | Function (计算规则) |
|------|----------------|---------------------|
| 存储方式 | 存在数据库 | 实时计算 |
| 谁来赋值 | 人工输入 或 外部系统写入 | 根据其他数据自动计算 |
| 典型例子 | 设备编号、安装日期 | 风险等级、在线状态 |

**常见错误**:
- ❌ 把"风险等级"设为Property,每次都要人工判断
  ✅ 应该用Function根据温度自动计算
- ❌ 把"设备名称"设为Function返回值
  ✅ 应该是Property,人工填写或导入

**记忆技巧**: 如果你能用"如果...那么..."描述这个数据,它应该是Function。
- "如果温度>90°C,那么风险等级=高" → Function ✅
- "设备编号就是EQ-001" → Property ✅

---

### Gotcha 3: Function设计得过于复杂
**问题**: Function的计算逻辑太复杂,包含多层嵌套判断、循环、外部API调用。

**为什么是问题**:
- Function会被频繁调用(界面显示、Automation判断),复杂逻辑会拖慢系统
- 复杂逻辑难以维护和调试
- 可能引入性能瓶颈

**建议**:
```
Function的计算逻辑应该:
✅ 在 < 100行代码以内
✅ 只依赖对象自身的Property
✅ 不调用外部API (除非性能可控)
✅ 避免复杂查询或大数据量遍历

如果计算逻辑很复杂:
→ 考虑是否应该拆分成多个Function
→ 考虑是否应该改用Automation (后台异步执行)
→ 考虑是否应该预先计算并存为Property
```

---

### Gotcha 4: 忘记设置Action的前置条件
**问题**: 用户可以在任何状态下执行任何操作,导致数据混乱。

**常见错误场景**:
```
❌ 已报废的设备还能"启动"
❌ 已完成的订单还能"取消"
❌ 设备温度正常时也能"紧急停机"
```

**正确做法** ✅:
```
每个Action都应该明确前置条件:

Action: 启动设备
前置条件:
  - 设备状态 = "停机" (不能是"已报废")
  - 设备是否在线 = "在线"
  - 设备温度 < 温度标准

Action: 取消订单
前置条件:
  - 订单状态 = "待处理" 或 "处理中"
  - 不能是"已发货" 或 "已完成"
```

---

### Gotcha 5: Automation的触发条件设置不当
**问题**: Automation触发太频繁 或 从不触发。

**触发太频繁的例子**:
```
❌ 触发条件: 温度 > 80°C
结果: 温度在80-90°C之间正常波动,每次波动都触发,一天触发100次

✅ 改进: 温度 > 90°C 且持续 > 5分钟
结果: 只有真正异常时才触发
```

**从不触发的例子**:
```
❌ 触发条件: Function "设备风险等级" == "高风险"
问题: Function名称写错了,实际是"设备温度风险等级"
结果: 条件永远不满足,Automation从不触发

✅ 测试方法:
1. 先手动构造触发条件 (如设置温度为100°C)
2. 观察Automation是否真的执行了
3. 检查日志确认触发原因
```

## ⚠️ 数据建模陷阱

### Gotcha 6: Property字段类型选错
**常见错误**:
| 错误做法 | 正确做法 | 影响 |
|---------|---------|------|
| 温度用String存"87.5°C" | 用Double存87.5 | 无法做数值比较和计算 |
| 日期用String存"2024-03-18" | 用Date类型 | 无法做日期范围查询 |
| 状态用String存任意文本 | 用Enum限定选项 | 用户可能输入错误值 |
| 编号用Integer存123 | 用String存"EQ-123" | 无法处理带前缀的编号 |

**记忆技巧**:
- 需要计算 → 数字类型 (Integer/Double)
- 需要日期比较 → Date/Timestamp
- 只有固定几个选项 → Enum
- 其他 → String

---

### Gotcha 7: 过度使用时序属性
**问题**: 把所有Property都设为时序属性,导致存储成本暴涨。

**判断标准**:
```
只对以下情况使用时序属性:
✅ 需要查看趋势图 (如温度变化曲线)
✅ 需要追溯历史值 (如"3天前库存是多少")
✅ 数据更新非常频繁 (如传感器数据)

不需要时序的情况:
❌ 设备编号 (固定不变)
❌ 设备名称 (很少变化)
❌ 所属车间 (偶尔变化,但不关心历史)
```

**成本对比**:
```
假设有1000台设备,每台每分钟上报一次温度:

非时序属性:
  - 存储量: 1000条记录 (每台设备只存当前值)

时序属性:
  - 存储量: 1000 × 60 × 24 × 30 = 4320万条记录/月
  - 存储成本高100倍以上
```

---

### Gotcha 8: 字段命名不清晰
**问题**: 使用技术化或含糊的字段名,业务人员看不懂。

**常见错误**:
```
❌ 不清晰的命名:
   - temp (什么的温度?)
   - status (哪个状态?)
   - flag1, flag2 (什么意思?)
   - data, value (太泛化)

✅ 清晰的命名:
   - 设备温度(°C)
   - 设备运行状态
   - 是否需要保养
   - 实时库存数量
```

**命名原则**:
- 使用完整的业务术语
- 包含单位 (温度(°C)、重量(kg))
- 避免缩写 (除非是业内通用的)
- 保持一致性 (统一用"编号"或"ID",不要混用)

## 🔧 实施阶段陷阱

### Gotcha 9: 没有充分测试就上线
**问题**: 直接在生产环境部署,出现问题后影响业务。

**必须测试的场景**:
```
✅ 正常场景:
   - 创建对象 → 成功
   - 执行操作 → 正常
   - 自动化触发 → 如预期

✅ 异常场景:
   - 输入不合法数据 → 是否正确拦截?
   - 不满足前置条件执行Action → 是否报错?
   - Function计算遇到空值 → 是否崩溃?

✅ 边界场景:
   - 温度刚好等于90°C → 是高风险还是低风险?
   - 同时有100个用户创建设备 → 系统是否正常?
   - Automation执行失败 → 是否有错误处理?
```

**测试环境建议**:
1. 先在测试环境部署
2. 用真实或接近真实的数据测试
3. 邀请3-5个实际用户试用1周
4. 收集反馈并优化
5. 才正式上线生产环境

---

### Gotcha 10: Automation没有限流保护
**问题**: Automation被过度频繁触发,淹没用户的通知渠道。

**常见场景**:
```
❌ 库存不足提醒:
   每次库存减少1件都发通知 → 一天1000条消息
   用户直接屏蔽所有通知

✅ 改进:
   - 同一商品24小时内最多提醒1次
   - 或者只在每天早上汇总一次性提醒
```

**限流配置建议**:
```
对于异常响应类Automation:
  - 同一对象N分钟内最多触发1次
  - 或设置冷却期: 触发后X分钟内不再触发

对于定时任务类Automation:
  - 合理设置执行频率 (不要设太频繁)
  - 批量处理而非逐个处理

对于流程编排类Automation:
  - 检测重复执行 (避免循环触发)
```

---

### Gotcha 11: 权限控制设置不当
**问题**: 给错误的角色授予了高危操作权限,或限制太严导致正常操作无法执行。

**常见错误**:
```
❌ 过于宽松:
   - 普通员工可以"删除所有数据"
   - 实习生可以"审批百万级采购"

❌ 过于严格:
   - 运维人员无法"变更设备状态"
   - 采购专员无法"创建采购订单"
```

**权限设计原则**:
```
✅ 最小权限原则:
   - 只给用户完成工作所需的最小权限
   - 高危操作 (删除、大额审批) 需要高权限

✅ 职责分离:
   - 申请和审批不能是同一人
   - 操作和验证应该分开

✅ 基于业务规则:
   - 采购专员可创建 ≤ 10万的订单
   - 采购经理可创建 ≤ 100万的订单
   - 超过100万需要总经理审批
```

---

### Gotcha 12: 忘记处理异常情况
**问题**: 只考虑了"一切顺利"的情况,没有处理失败场景。

**必须考虑的异常**:
```
Action执行失败:
  ✅ 设备通信超时 → 重试3次后提示用户
  ✅ 审批人不存在 → 提示"无效的审批人"
  ✅ 数据校验失败 → 明确提示哪个字段有问题

Function计算异常:
  ✅ 温度数据为空 → 返回"数据不足,无法判断"
  ✅ 除数为0 → 返回"计算错误"

Automation执行失败:
  ✅ 发送通知失败 → 记录日志,稍后重试
  ✅ 调用Action失败 → 通知管理员人工处理
```

## 💡 用户体验陷阱

### Gotcha 13: 操作按钮的文案不清晰
**问题**: 用户不知道点击按钮会发生什么。

**不清晰的例子**:
```
❌ 按钮文字:
   - "提交" (提交什么?)
   - "确定" (确定什么?)
   - "处理" (怎么处理?)

✅ 清晰的文字:
   - "提交报废申请"
   - "确定紧急停机"
   - "批准采购订单"
```

**确认弹窗要有足够信息**:
```
❌ "确定删除吗?"
✅ "确定删除设备'1#离心机'吗?删除后无法恢复。"
```

---

### Gotcha 14: 表单字段太多,用户填写负担重
**问题**: 创建对象时需要填写20+个字段,用户体验极差。

**优化方法**:
```
✅ 分清必填和选填:
   - 核心字段 (3-5个) → 必填
   - 其他字段 → 选填,可以后续补充

✅ 智能默认值:
   - 创建时间 → 自动填充当前时间
   - 创建人 → 自动填充当前用户
   - 状态 → 默认为"草稿"

✅ 关联数据自动带出:
   - 选择供应商后,自动填充联系人和地址
   - 选择产品后,自动填充价格和库存

✅ 分步表单:
   - 第一步: 基本信息 (3个字段)
   - 第二步: 详细信息 (5个字段)
   - 第三步: 附件和备注 (选填)
```

---

### Gotcha 15: 没有提供清晰的操作反馈
**问题**: 用户点击按钮后不知道是否成功。

**必须有的反馈**:
```
✅ 操作执行中:
   - 显示加载动画 + 提示文字: "正在停机,请稍候..."

✅ 操作成功:
   - 成功提示: "设备已成功停机"
   - 显示接下来可以做什么: "查看停机日志" "创建维修工单"

✅ 操作失败:
   - 失败提示: "停机失败: 设备通信超时"
   - 告知用户可以怎么做: "请检查设备网络连接,或联系技术支持"

✅ 操作需要等待:
   - 提示: "报废申请已提交,等待设备经理审批"
   - 提供跟踪链接: "查看审批进度"
```

## 🛠️ 维护阶段陷阱

### Gotcha 16: 没有记录配置变更历史
**问题**: 修改了配置但没有记录,出问题后不知道改了什么。

**建议做法**:
```
✅ 版本控制:
   - 每次修改配置文件后,标注版本号
   - 记录变更内容和原因
   - 保留历史版本以便回滚

✅ 变更日志示例:
   Version 1.1.0 (2024-03-18)
   - 修改: 温度风险判断从 > 85°C 改为 > 90°C
   - 原因: 用户反馈误报太多
   - 影响: Automation触发频率降低约50%

   Version 1.0.0 (2024-03-01)
   - 初始版本
```

---

### Gotcha 17: 一次性修改太多配置
**问题**: 同时修改了5个Function、3个Action、2个Automation,上线后出问题不知道是哪里导致的。

**建议**:
```
✅ 小步迭代:
   - 每次只修改1-2个配置项
   - 测试验证无误后再修改下一个
   - 这样出问题容易定位

❌ 大批量修改:
   - 一次性改10个配置
   - 上线后发现不工作
   - 不知道是哪个改错了
```

## 📝 文档陷阱

### Gotcha 18: 没有给业务人员提供操作手册
**问题**: 技术配置写得很完整,但业务人员不知道怎么用。

**必须提供的文档**:
```
✅ 用户操作手册:
   - 如何创建设备
   - 如何查看设备状态
   - 如何申请报废
   - 常见问题解答

✅ 业务流程说明:
   - 设备报废审批流程图
   - 保养计划管理流程
   - 异常响应流程

✅ 培训材料:
   - 快速上手视频 (5分钟)
   - 操作截图指南
   - 典型场景演示
```

---

### Gotcha 19: 配置文件和文档不一致
**问题**: 修改了配置但忘记更新文档,用户按文档操作却不工作。

**建议**:
```
✅ 同步更新:
   - 修改ontology-config.yaml时
   - 同时更新ontology-documentation.md
   - 检查用户手册是否需要更新

✅ 定期审查:
   - 每季度检查一次文档与实际配置是否一致
   - 修正发现的不一致
```

## ✅ 最佳实践总结

### 设计阶段
1. 先做MVP,再迭代 (不要追求第一版完美)
2. Property vs Function: 能自动计算的用Function
3. Function保持简单 (< 100行代码)
4. 每个Action必须有前置条件
5. Automation要有限流保护

### 数据建模
6. 字段类型选择: 需要计算→数字,固定选项→Enum
7. 谨慎使用时序属性 (只对需要趋势分析的字段使用)
8. 字段命名清晰 (用业务语言,包含单位)

### 实施阶段
9. 充分测试 (正常、异常、边界场景)
10. 权限控制要平衡 (不能太宽松也不能太严格)
11. 必须处理异常情况 (不要只考虑"一切顺利")

### 用户体验
12. 按钮文案清晰 (告诉用户会发生什么)
13. 表单字段精简 (3-5个必填,其他选填)
14. 提供明确的操作反馈 (成功/失败/等待)

### 维护阶段
15. 记录配置变更历史 (版本号 + 变更原因)
16. 小步迭代修改 (每次改1-2个配置)
17. 提供业务人员操作手册
18. 保持配置和文档一致

---

记住: **先把核心功能做好,再慢慢扩展。完成的70分方案,胜过永远在设计中的100分方案!**
