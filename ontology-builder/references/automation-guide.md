# 自动化规则指南（Automation Guide）

Automation（自动化规则）是系统的"神经系统"，它让系统能够自动响应事件、编排流程、定时执行任务，无需人工干预。

## 核心原则

### 什么是Automation？
Automation是"后台运行的智能监听器和编排器"：
- **事件驱动**：当特定事件发生时自动触发
- **流程编排**：串联多个Function和Action，形成完整业务流程
- **无需人工**：7×24小时自动运行

### Automation的价值
- 🚀 提升效率：原本需要人工检查的，现在自动完成
- 🛡️ 降低风险：关键异常立即响应，不依赖人工发现
- 📊 保证一致性：规则统一执行，避免人为疏漏

## Automation的三大类型

### 1. 异常响应（Event-Driven Response）

**业务场景**：当检测到异常或风险时，系统自动采取保护措施。

**示例1：高温自动停机**
```
触发条件：
  - Function "设备温度风险等级" 返回 "高风险"
  （即：设备温度 > 温度标准）

自动执行：
  Step 1: 调用 Action "紧急停机"
    - 硬件断电
    - 设备状态 → "停机"

  Step 2: 创建故障报告
    - 自动生成工单："设备 XX 高温停机"
    - 工单优先级 → "紧急"

  Step 3: 多渠道通知
    - 企业微信 → 设备负责人和运维经理
    - 短信 → 值班工程师
    - 邮件 → 设备经理（带详细日志）

  Step 4: 记录事件日志
    - 停机时间、温度数据、响应时长
```

**示例2：库存不足自动补货提醒**
```
触发条件：
  - Function "库存状态" 返回 "库存不足，需补货"
  （即：当前库存 < 安全库存下限）

自动执行：
  Step 1: 发送补货提醒
    - 企业微信 → 采购专员
    - 消息内容："【提醒】XX产品库存不足，当前库存XX，安全下限XX"

  Step 2: 自动创建采购申请草稿
    - 预填物品：当前不足的产品
    - 预填数量：安全库存上限 - 当前库存
    - 状态：草稿（等待采购专员确认）

  Step 3: 记录提醒历史
    - 避免短时间内重复提醒（24小时内只提醒一次）
```

**示例3：客户支付逾期自动催款**
```
触发条件：
  - Function "订单是否逾期" 返回 "逾期 X 天"
  - 订单状态 = "待付款"

自动执行：
  Step 1: 第一次逾期（1-3天）
    - 邮件催款 → 客户（温和提醒）

  Step 2: 第二次逾期（4-7天）
    - 邮件催款 → 客户（加强语气）
    - 短信催款 → 客户
    - 通知客户经理

  Step 3: 严重逾期（>7天）
    - 订单状态 → "异常"
    - 通知财务主管
    - 暂停该客户的新订单创建权限
```

### 2. 流程编排（Workflow Orchestration）

**业务场景**：当一个操作完成后，自动触发下一步操作，形成完整业务流程。

**示例1：设备报废审批流程**
```
触发事件：
  - Action "申请设备报废" 被执行

自动流程：
  Step 1: 创建审批工单
    - 工单类型："设备报废审批"
    - 审批人：设备经理
    - 附件：报废申请表、设备历史故障记录

  Step 2: 发送审批通知
    - 企业微信 → 设备经理
    - 消息："【待审批】XX 申请报废设备 YY，请审批"

  Step 3: 等待审批结果
    - 如果审批通过：
        → 设备状态更新为"已报废"
        → 创建资产处置工单
        → 通知财务部门更新资产账目
    - 如果审批拒绝：
        → 设备状态恢复为"停机"
        → 通知申请人拒绝原因

  Step 4: 记录审批历史
    - 审批人、审批时间、审批意见
```

**示例2：新供应商入驻流程**
```
触发事件：
  - Action "创建新供应商" 被执行

自动流程：
  Step 1: 发送欢迎邮件
    - 收件人：供应商邮箱
    - 内容：入驻指南、平台规则、联系方式

  Step 2: 创建入驻任务清单
    - 任务1：上传营业执照
    - 任务2：签署合作协议
    - 任务3：完成银行信息录入
    - 任务4：提交产品目录

  Step 3: 分配对接人
    - 根据供应商类型，自动分配采购专员
    - 发送通知给对接人

  Step 4: 7天后检查
    - 如果供应商未完成入驻任务：
        → 发送提醒邮件
        → 通知对接人跟进

  Step 5: 入驻完成
    - 当所有任务完成后：
        → 供应商状态 → "已认证"
        → 开通采购权限
```

**示例3：订单发货后续流程**
```
触发事件：
  - Action "确认发货" 被执行
  - 订单状态更新为"已发货"

自动流程：
  Step 1: 发送物流通知
    - 短信 → 客户手机
    - 邮件 → 客户邮箱
    - 内容：订单号、物流单号、预计送达时间

  Step 2: 更新库存
    - 扣减已发货商品的库存数量
    - 如果触发库存不足 → 启动补货Automation

  Step 3: 启动物流追踪
    - 定时查询物流状态（每6小时一次）
    - 当物流状态变为"已签收" → 更新订单状态为"已完成"

  Step 4: 自动确认收货（超时处理）
    - 如果发货后7天客户未确认收货：
        → 自动确认收货
        → 订单状态 → "已完成"
        → 发送评价邀请
```

### 3. 定时任务（Scheduled Jobs）

**业务场景**：按照固定时间周期执行的任务。

**示例1：每日设备健康报告**
```
触发时间：
  - 每天早上 8:00

自动执行：
  Step 1: 收集设备数据
    - 查询所有设备的昨日运行数据
    - 调用 Function "设备风险等级" 批量评估

  Step 2: 生成报告
    - 汇总统计：
        ○ 总设备数
        ○ 正常运行设备数
        ○ 异常设备数（高风险、故障）
        ○ TOP 5 高风险设备清单
    - 生成图表：设备状态分布、故障趋势

  Step 3: 发送报告
    - 邮件 → 设备经理、运维经理
    - 企业微信群 → 运维团队
    - 报告存档 → 系统文档库
```

**示例2：每周供应商绩效评估**
```
触发时间：
  - 每周一早上 9:00

自动执行：
  Step 1: 收集上周数据
    - 所有供应商的交付准时率
    - 所有供应商的质量问题次数
    - 所有供应商的响应速度

  Step 2: 批量计算绩效评分
    - 调用 Function "供应商综合评分"
    - 按评分排序，生成排行榜

  Step 3: 识别异常供应商
    - 评分 < 60分：标记为"需改进"
    - 连续两周评分下降：标记为"风险供应商"

  Step 4: 发送评估报告
    - 邮件 → 采购经理
    - 对于"需改进"的供应商 → 自动创建改进沟通任务

  Step 5: 更新供应商等级
    - 根据评分自动调整供应商分级（A/B/C/D级）
```

**示例3：每月数据归档**
```
触发时间：
  - 每月1号凌晨 2:00

自动执行：
  Step 1: 识别待归档数据
    - 查询所有状态为"已完成"且完成日期 > 90天的订单
    - 查询所有状态为"已关闭"的工单

  Step 2: 归档数据
    - 订单状态 → "已归档"
    - 工单状态 → "已归档"
    - 从主数据库移到归档数据库（可选）

  Step 3: 生成归档报告
    - 归档数量、归档数据量、释放的存储空间
    - 邮件 → 系统管理员

  Step 4: 清理临时数据
    - 删除超过6个月的系统日志
    - 删除超过1年的临时文件
```

## Automation设计技巧

### 技巧1：避免"自动化噪音"
不要过度通知，否则用户会忽略所有通知：
- ❌ 不好：每次库存变化都发通知
- ✅ 好：只有库存低于安全线时才通知，且24小时内最多通知一次

### 技巧2：设置熔断机制
如果Automation执行失败，应该有保护机制：
```
熔断规则示例：
  - 如果1小时内同一Automation失败 > 5次 → 暂停该Automation，通知管理员
  - 如果批量操作影响 > 100个对象 → 需要人工二次确认
```

### 技巧3：记录详细日志
Automation是后台运行的,必须有完整的日志：
```
日志应包含：
  - 触发时间
  - 触发条件（哪个Function返回了什么值）
  - 执行了哪些Action
  - 执行结果（成功/失败）
  - 发送了哪些通知
```

### 技巧4：提供手动触发和暂停
虽然是自动化,但应该提供手动控制：
```
管理功能：
  - 手动触发：紧急情况下立即执行某个Automation
  - 临时暂停：系统维护时暂停某些Automation
  - 历史查看：查看过去30天的执行记录
```

### 技巧5：先从简单规则开始,逐步优化
```
第一版（简单）：
  - 温度 > 90°C → 发送通知

第二版（优化）：
  - 温度 > 90°C 且持续 > 5分钟 → 发送通知
  （避免瞬时波动引起误报）

第三版（进阶）：
  - 温度 > 90°C 且持续 > 5分钟 → 发送通知
  - 温度 > 100°C → 立即紧急停机
  （分级响应）
```

## 常见问题

### Q1: Automation和Action有什么区别?
**A**:
- **Action**：由用户主动触发（点击按钮）
- **Automation**：由系统自动触发（事件驱动或定时）

### Q2: Automation可以调用其他Automation吗?
**A**: 可以,但要谨慎：
- ✅ 合理：报废审批通过 → 触发资产处置流程
- ❌ 危险：避免循环触发（A触发B,B触发A）

### Q3: Automation失败了会怎样?
**A**: 应该有完善的错误处理：
```
try:
  执行Automation
except 业务异常:
  记录日志,发送通知给相关人员
  根据配置决定是否重试（最多3次）
except 系统错误:
  记录错误日志,发送告警给系统管理员
  暂停该Automation,等待人工处理
```

### Q4: 一个对象应该有多少个Automation?
**A**: 通常2-8个：
- 关键异常响应：1-3个
- 流程编排：1-2个
- 定时任务：0-3个

### Q5: 如何测试Automation?
**A**: 三种测试方法：
1. **手动触发测试**：提供测试按钮,模拟触发条件
2. **沙盒测试**：在测试环境验证,避免影响生产数据
3. **干运行模式**：执行逻辑但不真正修改数据,只输出日志

## Automation命名建议

**推荐做法** ✅：
- 描述"什么时候做什么"：
  - "高温时自动停机"
  - "订单发货后通知客户"
  - "每日生成设备健康报告"
- 清晰明确：
  - ✅ "库存不足时自动补货提醒"
  - ❌ "库存管理自动化"

**避免的做法** ❌：
- 过于技术：`auto_shutdown_v2`
- 过于简略：`自动化1` `规则A`
- 含糊不清：`设备监控` `数据处理`

## Automation的完整示例

让我们看一个完整的案例,展示Property → Function → Automation → Action的配合：

```
【场景】设备温度监控与自动保护

1. Property（数据感知）
   - 设备温度：87.5°C（时序属性,实时更新）
   - 温度标准：90°C
   - 设备状态：运行中

2. Function（智能判断）
   名称：设备温度风险等级
   逻辑：
     如果 设备温度 > 温度标准：返回 "高风险"
     否则如果 设备温度 > 温度标准 × 0.8：返回 "中风险"
     否则：返回 "低风险"
   当前结果：中风险（87.5 > 90×0.8=72）

3. Automation（后台监听）
   名称：高温自动保护机制
   触发条件：Function "设备温度风险等级" 返回 "高风险"
   执行流程：
     Step 1: 调用 Action "紧急停机"
     Step 2: 创建故障报告工单
     Step 3: 发送多渠道通知
     当前状态：未触发（因为当前是"中风险"）

4. Action（执行干预）
   名称：紧急停机
   效果：
     - 硬件断电
     - 设备状态 → "停机"
     - 记录停机日志
   当前状态：待命（只有Automation触发才会执行）

【当温度升高到95°C时】
  Property更新 → Function重新计算返回"高风险"
  → Automation捕捉到变化 → 自动调用Action
  → 设备停机,发送通知 → 保护生产安全 ✅
```

这就是Palantir本体四要素的完美协作！
