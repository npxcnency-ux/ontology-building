# 计算规则指南（Function Guide）

计算规则（Function）是"只读的大脑"，它根据已有数据自动计算结果，但不会修改任何数据。

## 核心原则

### 什么是Function？
Function像是一个"自动计算器"或"智能判断器"：
- **输入**：读取对象的属性数据
- **处理**：按照你定义的逻辑计算或判断
- **输出**：返回计算结果（但不写入数据库）

### Function的特点
- ✅ 只读：不修改任何数据
- ✅ 无副作用：运行多少次都不会改变系统状态
- ✅ 可复用：可以在界面、自动化、Action中多次调用

## Function的三大类型

### 1. 风险判断规则（Risk Assessment）

**业务场景**：自动评估业务风险等级

**示例1：设备温度风险判断**
```
规则名称：设备温度风险等级
输入数据：
  - 当前温度
  - 温度标准（警戒线）
计算逻辑：
  如果 当前温度 > 温度标准：
    返回 "高风险"
  否则如果 当前温度 > 温度标准 × 0.8：
    返回 "中风险"
  否则：
    返回 "低风险"
```

**示例2：供应商信用评估**
```
规则名称：供应商信用等级
输入数据：
  - 延迟交付次数
  - 质量问题次数
  - 合作年限
计算逻辑：
  信用分数 = 100 - (延迟次数 × 5) - (质量问题 × 10) + (合作年限 × 2)
  如果 信用分数 >= 80：返回 "优秀"
  如果 信用分数 >= 60：返回 "良好"
  如果 信用分数 >= 40：返回 "一般"
  否则：返回 "差"
```

### 2. 状态推断规则（Status Inference）

**业务场景**：根据其他数据自动推断对象的状态

**示例1：设备在线状态推断**
```
规则名称：设备是否在线
输入数据：
  - 最后心跳时间
  - 当前系统时间
计算逻辑：
  时间差 = 当前时间 - 最后心跳时间
  如果 时间差 < 5分钟：
    返回 "在线"
  否则：
    返回 "离线"
```

**示例2：订单逾期判断**
```
规则名称：订单是否逾期
输入数据：
  - 订单状态
  - 预计交付日期
  - 当前日期
计算逻辑：
  如果 订单状态 == "已完成"：
    返回 "已完成，无逾期"
  如果 当前日期 > 预计交付日期：
    返回 "逾期 X 天"
  否则：
    返回 "未逾期"
```

**示例3：库存预警**
```
规则名称：库存状态
输入数据：
  - 当前库存数量
  - 安全库存下限
  - 最大库存上限
计算逻辑：
  如果 库存 < 安全下限：返回 "库存不足，需补货"
  如果 库存 > 最大上限：返回 "库存过剩，需促销"
  否则：返回 "库存正常"
```

### 3. 业务计算公式（Business Metrics）

**业务场景**：自动计算业务指标、KPI

**示例1：设备利用率**
```
规则名称：设备利用率
输入数据：
  - 实际运行时长（小时）
  - 计划运行时长（小时）
计算逻辑：
  利用率 = (实际运行时长 / 计划运行时长) × 100%
  返回：利用率（百分比）
```

**示例2：供应商综合评分**
```
规则名称：供应商综合评分
输入数据：
  - 质量评分（0-100）
  - 交付准时率（0-100）
  - 价格竞争力（0-100）
  - 响应速度（0-100）
计算逻辑：
  综合评分 = 质量评分 × 0.4
            + 交付准时率 × 0.3
            + 价格竞争力 × 0.2
            + 响应速度 × 0.1
  返回：综合评分（保留2位小数）
```

**示例3：客户生命周期价值（LTV）**
```
规则名称：客户生命周期价值
输入数据：
  - 平均订单金额
  - 年购买频次
  - 客户关系年限
计算逻辑：
  LTV = 平均订单金额 × 年购买频次 × 客户关系年限
  返回：LTV（元）
```

## Function设计技巧

### 技巧1：先从简单规则开始
不要一开始就设计复杂的多重判断。先从最核心的1-2条规则开始：
- ❌ 复杂：考虑10种设备状态、5种温度区间、3种车间类型...
- ✅ 简单：先判断"温度是否超标"，后续再扩展

### 技巧2：输出结果要明确
Function的返回值应该清晰易懂：
- ✅ 好的输出："高风险" "需要补货" "逾期3天"
- ❌ 不好的输出："1" "true" "状态A"

### 技巧3：避免过度依赖外部数据
Function应该主要依赖对象自身的属性，避免复杂的跨对象查询：
- ✅ 简单：读取本对象的"温度"和"温度标准"
- ❌ 复杂：查询所有相同类型设备的平均温度，再与行业标准比较...

### 技巧4：为关键判断添加容错逻辑
```
如果 温度数据为空 或 温度标准未设置：
  返回 "数据不足，无法判断"
否则：
  执行正常判断逻辑
```

## 常见问题

### Q1: Function和Property有什么区别？
**A**:
- **Property（属性）**：存储在数据库中的固定值，如"设备编号 = EQ-001"
- **Function（计算规则）**：每次调用时实时计算，如"风险等级 = 高"（根据当前温度计算）

### Q2: Function可以修改数据吗？
**A**: 不可以！Function是只读的。如果需要修改数据，必须用Action（操作）。

### Q3: Function的计算会很慢吗？
**A**: 简单的判断和计算非常快（毫秒级）。如果Function涉及复杂查询或大量数据，可能会慢一些，但这是Palantir平台会自动优化的。

### Q4: 我可以在Function里调用外部API吗？
**A**: 理论上可以，但不推荐。Function应该是"纯计算"，复杂的外部调用应该放在Automation中。

### Q5: 我需要多少个Function？
**A**: 通常一个对象有3-8个Function就足够了。太多会增加维护成本，太少会缺少智能判断能力。

## Function命名建议

**推荐做法** ✅：
- 使用问句或结果描述：
  - "设备是否过热"
  - "供应商信用等级"
  - "订单逾期天数"
- 包含业务含义，而非技术实现：
  - ✅ "客户流失风险"
  - ❌ "customer_churn_prediction_model_v2"

**避免的做法** ❌：
- 过于技术：`calculate_temp_risk()`
- 过于简略：`check1` `rule_a`
- 含糊不清：`状态判断` `数据计算`

## Function与Action/Automation的关系

```
┌─────────────┐
│  Property   │ (数据字段：设备温度 = 95°C)
└──────┬──────┘
       │ 读取
       ↓
┌─────────────┐
│  Function   │ (计算规则：温度风险 = 高)
└──────┬──────┘
       │ 触发
       ↓
┌─────────────┐
│ Automation  │ (自动化：检测到高风险)
└──────┬──────┘
       │ 执行
       ↓
┌─────────────┐
│   Action    │ (操作：紧急停机)
└─────────────┘
```

Function是连接"数据感知"和"自动响应"的桥梁！
