# CLAUDE.md - Ontology Builder Skill 项目记忆

这是Ontology Builder Skill项目的长期记忆文档。记录了设计决策、经验教训、最佳实践和重要约定。

---

## 📋 项目概览

### 项目目标
创建一个面向**企业白领**（非技术人员）的交互式本体构建skill，通过友好的对话将业务概念转化为Palantir平台可用的本体配置。

### 核心特点
- 🗣️ **对话式交互**：逐步引导，不是一次性表单
- 📚 **业务语言**：完全避免技术术语
- 🎯 **强制完整性**：5阶段Pipeline确保本体四要素完整
- 💡 **内置示例**：真实业务场景参考

### 关键指标
- **总代码量**: 3,968行
- **核心文件**: 12个
- **参考指南**: 6个 (2,002行)
- **完整示例**: 1个 (设备监控，500行)
- **覆盖陷阱**: 19个
- **质量检查体系**: 3级门禁（P0/P1/P2，共20项检查）

## 🎯 设计决策记录

### 决策1: 使用 Inversion + Pipeline 混合模式

**背景**: 需要既理解用户需求，又确保配置完整性

**选择的方案**: Inversion + Pipeline 混合模式
- **Inversion**: Agent先理解业务需求（阶段一）
- **Pipeline**: 强制执行5阶段流程，每阶段有检查点

**备选方案及为何放弃**:
- ❌ 纯Generator模式：无法处理复杂的用户需求探索
- ❌ 纯Pipeline模式：太机械，用户体验差
- ❌ 纯Inversion模式：缺少流程控制，容易遗漏配置

**结果**: 成功平衡了灵活性和严格性

---

### 决策2: 完全避免技术术语

**背景**: 目标用户是企业白领，不懂技术术语

**翻译表**:
| 技术术语 | 业务语言 | 理由 |
|---------|---------|------|
| Property | 数据字段 | 白领理解"字段" |
| Function | 计算规则 | 描述"自动计算"的本质 |
| Action | 操作按钮 | 直观对应界面元素 |
| Automation | 自动化规则 | 描述"自动执行"的特性 |
| Object Type | 业务对象 | 更贴近业务场景 |
| Time Series | 时序属性/历史追踪 | 说明"为什么需要" |

**关键原则**:
- ✅ 用"是什么"描述，不用"叫什么"
- ✅ 每个概念提供2-3个具体例子
- ✅ 避免缩写（EQ-001 ✓，但要解释"设备编号"）

---

### 决策3: 5阶段流程而非3阶段

**背景**: Palantir本体有4个要素，需要合理划分阶段

**选择的方案**: 5阶段
1. 理解业务对象 (5分钟)
2. 定义数据字段 (10分钟) → Property
3. 定义计算规则 (10分钟) → Function
4. 定义操作按钮 (10分钟) → Action
5. 定义自动化规则 (15分钟) → Automation

**备选方案及为何放弃**:
- ❌ 3阶段（业务理解+数据建模+操作设计）：Action和Automation容易被混淆
- ❌ 4阶段（对应四要素）：缺少"理解业务"的前置阶段
- ❌ 7阶段：过于碎片化，用户容易失去耐心

**结果**: 每个阶段时长适中，用户不会疲劳

---

### 决策4: 只提供1个完整示例（而非3个）

**背景**: 最初计划3个示例（设备、供应商、订单），但实现时只完成了1个

**选择的方案**: 专注做好1个高质量示例
- ✅ 设备监控：12个Property、3个Function、5个Action、4个Automation
- ✅ 包含完整的业务背景、使用场景、价值说明

**经验教训**:
- ⚠️ **过度承诺陷阱**: 在SKILL.md中写了3个例子，实际只有1个 → Bug #1
- ✅ **修复方式**: 修改SKILL.md只承诺现有的例子
- 📝 **教训**: 先完成1个验证质量，再承诺更多

**后续计划**:
- Phase 2可以增加供应链、CRM示例
- 但必须先完成再更新SKILL.md

---

### 决策5: 双输出设计（YAML + Markdown）

**背景**: 技术人员需要YAML配置，业务人员需要易读文档

**选择的方案**: 同时生成两个文件
- `ontology-config.yaml`: 标准Palantir配置（给开发人员部署）
- `ontology-documentation.md`: 业务说明文档（给业务人员理解）

**关键点**:
- ✅ 两个文件从同一份用户输入生成，确保一致性
- ✅ 文档包含"为什么"（业务场景），配置只有"是什么"（技术参数）
- ✅ 避免了Gotcha #19："配置文件和文档不一致"

### Gotcha #2: 没有使用 brainstorming skill

**问题**: 这是创造性工作（创建新skill），应该先brainstorm，但直接开始实现了

**发现时间**: 2026-03-19（用户提问）

**影响**:
- 跳过了探索多种设计方案的机会
- 用户没有参与设计决策
- 很多设计选择是我自己决定的（可能不是最优）

**应该的流程**:
```
用户: "创建skill..."
  ↓
识别到创造性工作
  ↓
调用 superpowers:brainstorming
  ↓
探索: Inversion vs Pipeline vs 其他模式？
探索: 3阶段 vs 5阶段？
探索: 单输出 vs 双输出？
  ↓
与用户讨论并确定方案
  ↓
然后再开始实现
```

**为什么这次侥幸成功**:
- 用户提供了明确的输入（3个参考文档、目标用户）
- 我在执行中进行了"隐式brainstorming"
- 但缺少了用户的反馈循环

**教训**:
- ⚠️ **即使需求看起来明确，也要先brainstorm**
- ⚠️ **创建skill属于"creating features"，必须触发brainstorming**
- ⚠️ **让用户参与设计决策，而非自己全部决定**

**补救措施**:
- 如果用户不满意现有设计，可以重新brainstorm
- 下次创建类似项目，必须先用brainstorming skill

### Gotcha #3: 时序属性的成本被低估

**问题**: 容易把所有Property都设为时序属性，导致存储成本暴涨

**预防措施**:
- ✅ 在 `property-guide.md` 明确说明成本对比
- ✅ 提供判断标准：只对"需要查看趋势"的字段使用
- ✅ 示例：
  - ✅ 设备温度（需要趋势图）→ 时序 ✓
  - ❌ 设备编号（固定不变）→ 非时序 ✓
  - ❌ 设备名称（很少变化）→ 非时序 ✓

**计算示例**:
```
1000台设备 × 60分钟 × 24小时 × 30天 = 4320万条记录/月
存储成本高100倍以上
```

### Gotcha #4: Property vs Function 混淆

**问题**: 不知道某个数据应该存为Property还是用Function计算

**判断标准**:
| 特征 | Property | Function |
|------|----------|----------|
| 谁赋值 | 人工输入 或 外部系统写入 | 自动计算 |
| 存储方式 | 存在数据库 | 实时计算 |
| 典型例子 | 设备编号、安装日期 | 风险等级、在线状态 |

**记忆技巧**: 如果能用"如果...那么..."描述，就是Function
- "如果温度>90°C，那么风险=高" → Function ✅
- "设备编号就是EQ-001" → Property ✅

---

### Gotcha #5: Automation触发太频繁

**问题**: 没有限流保护，导致用户被通知淹没

**预防措施**:
- ✅ 在 `automation-guide.md` 强调限流配置
- ✅ 示例：
  ```yaml
  rateLimiting:
    maxExecutionsPerHour: 1  # 每小时最多1次
    cooldownMinutes: 60      # 两次执行间隔至少60分钟
  ```
- ✅ 反例：❌ 每次温度变化都发通知 → 一天1000条
- ✅ 正例：✅ 同一设备24小时内最多提醒1次

---

### Gotcha #6: Action缺少前置条件

**问题**: 用户可以在任何状态执行任何操作，导致数据混乱

**预防措施**:
- ✅ 在 `action-guide.md` 强调前置条件的重要性
- ✅ 每个Action都必须明确前置条件
- ✅ 示例：
  ```yaml
  # ❌ 错误：已报废的设备还能启动
  Action: 启动设备
  前置条件: 无

  # ✅ 正确
  Action: 启动设备
  前置条件:
    - 设备状态 = "停机" (不能是"已报废")
    - 设备是否在线 = "在线"
  ```

## 🎓 最佳实践总结

### 编写Skill的黄金法则

#### 1. 先Brainstorm，再实现
```
创造性工作（创建skill、添加功能）
→ 必须先调用 superpowers:brainstorming
→ 探索多种方案
→ 让用户参与决策
→ 然后再实现
```

#### 2. 先实现，再承诺
```
❌ 错误流程:
  1. 在SKILL.md写"我可以展示3个例子"
  2. 实际只实现1个
  3. 用户选择不存在的例子 → 报错

✅ 正确流程:
  1. 先完成1个高质量示例
  2. 验证可用
  3. 然后在SKILL.md承诺这1个
  4. 想加更多？重复步骤1-3
```

#### 3. 文档与实现同步
```
修改代码/添加文件
  ↓
立即更新相关文档
  ↓
运行验证脚本
  ↓
确保一致性
```

#### 4. 渐进式加载，保持context简洁
```
# ✅ 好的做法
阶段二 → 只加载 property-guide.md
阶段三 → 只加载 function-guide.md

# ❌ 不好的做法
开始时 → 一次性加载所有5个reference文件
结果: context爆炸，信息过载
```

#### 5. 用业务语言，提供大量例子
```
# ❌ 技术语言
"Property的dataType可以是String、Integer、Date..."

# ✅ 业务语言
"这个数据字段应该存文字还是数字？
- 存文字: 设备名称（如'1#离心机'）
- 存数字: 设备温度（如87.5）
- 存日期: 安装日期（如2024-03-15）"
```

#### 6. 大文件必须拆分，保持主文件精简
```
# ❌ 错误做法
SKILL.md包含所有详细内容 → 469行 → 难以维护

# ✅ 正确做法
SKILL.md: 只包含核心流程（300行）
references/: 详细规则独立文件（6个文件，2,002行）

# 拆分原则
- 主文件：流程逻辑 + 引用指令
- reference文件：详细规则 + 示例 + 模板
- 渐进式加载：需要时才加载
```

### 删除中间态文件

**原则**: 项目完成后，及时删除中间态报告文件，只保留最终文档。

**为什么要删除**:
- 避免信息冗余（中间报告的内容已整合到最终文档）
- 保持项目结构清晰
- 减少维护负担
- 符合"Don't State the Obvious"原则

**删除流程**:

1. **识别中间态文件**
   - Task完成报告（TASK_X_COMPLETION_REPORT.md）
   - 阶段性验证报告（SKILL_VERIFICATION_REPORT.md）
   - Bug修复记录（BUG_FIX_RECORD.md）- 如果已整合到CLAUDE.md
   - 项目摘要（PROJECT_SUMMARY.md）- 如果有更完整的最终报告

2. **确认信息已整合**

   删除前必须确认信息去向：

   | 删除的文件 | 信息去向 |
   |-----------|---------|
   | TASK_X_COMPLETION_REPORT.md | 整合到PROJECT_COMPLETION_REPORT.md或CLAUDE.md |
   | BUG_FIX_RECORD.md | 整合到CLAUDE.md的"已知陷阱"部分 |
   | SKILL_VERIFICATION_REPORT.md | 整合到PROJECT_COMPLETION_REPORT.md |
   | PROJECT_SUMMARY.md | 被更完整的PROJECT_COMPLETION_REPORT.md替代 |

3. **执行删除**
   ```bash
   # 删除中间态文件
   rm -f TASK_*_REPORT.md
   rm -f BUG_FIX_RECORD.md
   rm -f SKILL_VERIFICATION_REPORT.md
   rm -f PROJECT_SUMMARY.md
   ```

4. **验证结果**
   ```bash
   # 检查是否还有不需要的文件
   find . -name "*REPORT.md" -o -name "*RECORD.md" | grep -v "PROJECT_COMPLETION_REPORT\|CLEANUP_REPORT"
   ```

5. **更新CLEANUP_REPORT.md**
   - 记录删除了哪些文件
   - 说明每个文件的信息去向
   - 用于未来追溯

**保留的文件**:
- ✅ PROJECT_COMPLETION_REPORT.md（最终完成报告）
- ✅ CLAUDE.md（项目记忆）
- ✅ README.md（项目说明）
- ✅ docs/CLEANUP_REPORT.md（清理记录）
- ✅ docs/TESTING.md（TDD测试文档）
- ✅ docs/VERIFICATION.md（验证清单）

**实际案例**（本项目）:
```bash
# 2026-03-19执行的清理
删除了6个中间态文件：
- TASK_7_FIX_REPORT.md → 内容已整合到CLAUDE.md Gotcha #3
- TASK_9_COMPLETION_REPORT.md → 内容已整合到PROJECT_COMPLETION_REPORT.md
- TDD_COMPLETION_REPORT.md → 内容已整合到docs/TESTING.md
- SKILL_VERIFICATION_REPORT.md → 内容已整合到PROJECT_COMPLETION_REPORT.md
- BUG_FIX_RECORD.md → 内容已整合到CLAUDE.md Gotcha #1
- PROJECT_SUMMARY.md → 被PROJECT_COMPLETION_REPORT.md替代

结果: markdown文件从21个减少到10个（-52%）
```

**关键原则**:
- ✅ 先整合信息，再删除文件
- ✅ 在CLEANUP_REPORT.md记录删除历史
- ✅ 保留核心文档（CLAUDE.md, PROJECT_COMPLETION_REPORT.md, README.md）
- ❌ 不要删除skill核心文件（SKILL.md, references/, examples/）
- ❌ 不要删除项目文档（docs/目录下的文件）

## 🎯 验收标准体系

### 设计决策

**决策时间**: 2026-03-20

**背景**: ontology-builder skill缺少系统化的验收标准，无法评估生成配置的质量和业务价值。

**选择的方案**: 两阶段门禁模型（方案C）

#### 为什么选择两阶段门禁？

**备选方案回顾**:
- 方案A："价值金字塔"模型 - 分层递进，但业务价值难以量化
- 方案B："加权评分卡"模型 - 量化清晰，但评分有主观性
- **方案C："两阶段门禁"模型** - ✅ 选中

**选择理由**:
1. **职责分离清晰**: 机器检查客观规则，人判断主观价值
2. **效率最高**: 自动门禁快速过滤明显缺陷，人工只评审有潜力的配置
3. **符合优先级**: 业务价值(50%) > 可用性(30%) > 完整性(20%)
4. **可渐进式演进**: 先实现自动门禁，再建立业务评审流程

---

### 两阶段门禁详解

#### 阶段1：自动门禁（Auto Gate）

**目的**: 快速过滤技术缺陷
**执行时机**: SKILL.md生成配置时自动执行
**检查内容**:

| 级别 | 名称 | 检查项数 | 行为 |
|------|------|---------|------|
| P0 | Blocker | 8项 | 阻止生成配置 |
| P1 | Warning | 8项 | 警告但允许继续 |
| P2 | Info | 4项 | 仅提示优化建议 |

**P0级检查项（必须通过）**:
1. Property ≥ 3个
2. Action ≥ 1个
3. 定义了primaryKey
4. 所有Action有permissions
5. 高危Action有confirmationRequired
6. Automation引用的Function存在
7. Automation引用的Action存在
8. 无循环触发风险

**P1级检查项（强烈建议）**:
1. 数字字段包含单位
2. 必填字段 ≤ 7个
3. 字段使用业务语言
4. 明确时序属性配置
5. Function逻辑 < 100行
6. Automation有限流保护
7. Action有前置条件
8. Function有异常处理

**P2级检查项（可选优化）**:
1. 生成了业务文档
2. 填写了完整元数据
3. Action类型规范
4. Function有usageScenarios

---

#### 阶段2：业务门禁（Business Gate）

**目的**: 评估业务价值
**执行时机**: 团队评审会议（人工）
**评审内容**: 5个核心问题（总分80分）

| 问题 | 分值 | 评审工具 |
|------|------|---------|
| 1. 业务痛点清晰度 | 20分 | 5W1H痛点挖掘表 |
| 2. 解决方案匹配度 | 20分 | 功能价值矩阵 |
| 3. 效率提升量化 | 15分 | ROI前后对比表 |
| 4. 用户采用可行性 | 15分 | 用户画像障碍分析 |
| 5. 可扩展性与演进 | 10分 | 未来场景推演 |

**评级标准**:
- 70-80分 → ⭐⭐⭐⭐⭐ High Value（优先部署）
- 50-69分 → ⭐⭐⭐ Medium Value（正常部署）
- <50分 → 🤔 Low Value（重新评估）

### 关键原则

1. **自动门禁只检查客观规则**
   - ✅ 数量、格式、引用完整性
   - ❌ 命名是否"好听"、设计是否"优雅"

2. **业务门禁专注价值判断**
   - ✅ 是否解决真实痛点
   - ✅ 是否有人愿意用
   - ✅ 是否值得投入资源

3. **失败必须给建议**
   - 不能只说"有问题"
   - 要告诉用户"怎么修复"

4. **评分必须有依据**
   - 提供评审工具（表格、清单）
   - 不能拍脑袋打分

## 🎓 总结：最重要的4条教训

### 1. 先Brainstorm，再实现
即使需求看起来明确，也要先用brainstorming skill探索方案，让用户参与决策。

### 2. 先实现，再承诺
不要在文档中承诺还未实现的功能。先完成1个高质量功能，验证后再承诺更多。

### 3. 文档与实现同步
每次修改代码/添加文件后，立即更新相关文档。定期运行验证脚本确保一致性。

### 4. 大文件必须拆分，保持主文件精简（新增 2026-03-20）
**主文件过长的危害**:
- 核心流程被细节淹没，难以理解
- 修改成本高（需要在大文件中定位）
- 违反"保持精简"原则
- 与渐进式加载模式不一致

**拆分原则**:
- 主文件（SKILL.md）：只包含核心流程逻辑 + 引用指令
- 参考文件（references/）：详细规则 + 示例 + 模板
- 渐进式加载：需要时才加载

**实际案例**（Step 4重构）:
```
修改前: SKILL.md包含所有检查项详解 → 469行
修改后: SKILL.md只有检查概述 → 300行（-36%）
详细规则: 移到quality-check-guide.md → 382行
结果: 主文件精简，维护更容易
```

### 5. **Skills也要TDD！**（新增 2026-03-19）
**No Skill Without Test First.** 这是writing-skills的铁律，和代码的TDD一样严格：
- RED: 运行baseline测试（没有skill时agent怎么做）
- GREEN: 验证skill解决了baseline问题
- REFACTOR: 找到新rationalization并修复
- 没有测试就部署skill = 违反TDD，必须重来
