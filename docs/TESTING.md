## 🔍 推断的Baseline行为

### 没有skill时，Claude可能会：

#### 1. 语言使用问题

**Baseline (错误)**:
```
我来帮你构建Palantir本体。首先，我们需要定义Object Type的Properties。
请告诉我：
1. 你需要哪些Property fields？
2. 每个Property的dataType是什么（String, Integer, Date等）？
3. 哪些Property需要配置为Time Series？
```

**问题**:
- ❌ 使用技术术语（Property, dataType, Time Series）
- ❌ 假设用户理解这些概念
- ❌ 一次问太多问题

**期望 (有skill后)**:
```
我会帮你一步步构建业务对象的配置。

首先，你想管理什么业务对象？
例如：供应商、客户、订单、设备、产品、员工等
```

**期望行为**:
- ✅ 使用"业务对象"而非"Object Type"
- ✅ 提供具体例子
- ✅ 一次只问一个问题

---

#### 2. 流程结构问题

**Baseline (错误)**:
```
[直接开始问各种问题，没有明确的阶段]

1. 对象名称是什么？
2. 需要哪些属性？
3. 需要什么操作？
4. 有自动化需求吗？
5. 生成YAML还是JSON？
```

**问题**:
- ❌ 没有阶段划分
- ❌ 问题顺序随意
- ❌ 没有检查点
- ❌ 可能遗漏某些要素（如Function）

**期望 (有skill后)**:
```
阶段一：理解业务对象
- Q1: 对象识别
- Q2: 业务场景
- Q3: 关键人员
- 检查点: 等待用户确认 ✓

阶段二：定义数据字段
- ...
- 检查点: 等待用户确认 ✓

[依次类推5个阶段]
```

**期望行为**:
- ✅ 清晰的5阶段结构
- ✅ 每个阶段有明确目标
- ✅ 每个阶段后有检查点
- ✅ 强制完成四要素

---

#### 3. 完整性问题

**Baseline (错误)**:
```
好的，我为你生成了设备管理的本体配置：
```

```
**配置摘要：**

**生成的文件：**
1. ontology-config.yaml - 完整的技术配置
2. ontology-documentation.md - 业务说明文档
```

**期望行为**:
- ✅ 强制包含四要素

```
Claude: "没关系，我来帮你整理。设备对象通常需要记录：

**基本信息**：

- 设备名称（如 1#离心机）
- 所属车间（如 A栋2楼）

**状态信息**：
```

```
- 操作按钮（用户交互）✓
```

- ✅ 提供大量例子帮助理解
- ✅ 不允许跳过关键要素

**Baseline**: 只生成YAML配置

### 问题5: 遇阻就放弃
**Baseline**: 用户说"不知道"就无法继续
**Skill解决**: 提供大量例子，引导用户思考

### 问题6: 允许跳过关键步骤

## ✅ Baseline测试结论

### 没有skill时的主要问题：

1. ⚠️ **语言障碍**: 技术术语让非技术用户困惑
2. ⚠️ **结构缺失**: 没有清晰的流程，容易遗漏
3. ⚠️ **完整性差**: 可能只生成部分要素
4. ⚠️ **输出单一**: 只有技术配置，缺少业务文档
5. ⚠️ **易受阻**: 用户不懂就卡住
6. ⚠️ **可跳过**: 允许跳过关键步骤

### Skill的核心价值：

1. ✅ **业务语言**: 完全避免技术术语
2. ✅ **强制流程**: 5阶段+检查点
3. ✅ **保证完整**: 四要素缺一不可
4. ✅ **双输出**: YAML + 文档
5. ✅ **持续引导**: 大量例子帮助理解
6. ✅ **防止跳过**: 解释重要性，不允许省略

## 🔵 REFACTOR阶段：需要改进的地方

### 潜在的Rationalization

虽然skill设计良好，但仍可能出现以下rationalization：

#### Rationalization 1: "阶段太多了，能不能合并？"
**当前Counter**: "不要跳过任何步骤"
**评估**: ⚠️ 不够强 - 没有解释"为什么需要5个阶段"
**建议改进**: 添加明确的理由

```markdown
**为什么需要5个阶段？**

如果合并阶段：
- ❌ 一次性问太多问题 → 用户信息过载
- ❌ 混淆不同类型的配置 → 容易出错
- ❌ 没有检查点 → 发现错误时已经太晚

5个阶段的好处：
- ✅ 每次只关注一个方面 → 易于理解
- ✅ 循序渐进 → 自然的思考流程
- ✅ 有检查点 → 及时发现和修正问题
```

#### Rationalization 2: "我可以直接看例子然后自己写吗？"
**当前Counter**: 无明确counter
**评估**: ⚠️ 缺失 - skill应该说明"为什么需要对话而不是照抄例子"
**建议改进**: 在引导用户加载参考资料部分添加

```markdown
> 你想先看例子吗？还是直接开始？
>
> **注意**: 例子可以参考，但不能直接照抄，因为：
> - 每个企业的业务场景不同
> - 字段名称、规则逻辑需要定制
> - 通过对话，我能帮你思考遗漏的地方
```

#### Rationalization 3: "Function看起来不重要，我跳过了"
**当前Counter**: "不要跳过任何步骤"（过于general）
**评估**: ⚠️ 不够具体 - 没有解释"为什么Function重要"
**建议改进**: 在阶段三添加明确说明

```markdown
## 阶段三：定义计算规则

**为什么Function不能跳过？**

没有Function的后果：
- ❌ 风险需要人工判断 → 反应慢，容易遗漏
- ❌ 状态需要人工更新 → 数据不及时
- ❌ 指标需要人工计算 → 效率低，易出错

有了Function之后：
- ✅ 系统自动判断风险 → 实时响应
- ✅ 状态自动推断 → 数据永远最新
- ✅ 指标自动计算 → 准确高效

**Function是Automation的大脑** - 没有Function，Automation无法判断何时触发。
```

#### Rationalization 4: "我只要YAML配置，不需要Markdown文档"
**当前Counter**: 无明确counter
**评估**: ⚠️ 缺失
**建议改进**: 在输出阶段添加说明

```markdown
## 输出阶段：生成配置文件

**为什么生成两个文件？**

1. **ontology-config.yaml** (技术配置)
   - 给开发人员部署用
   - 标准Palantir格式
   - 可直接导入平台

2. **ontology-documentation.md** (业务文档)
   - 给业务人员理解用
   - 说明业务场景和规则
   - 未来维护时的参考

**两者必须一致** - 只有YAML容易出现"配置和实际业务不符"的问题。
```

```markdown
**注意**: 例子可以参考，但不能直接照抄，因为：
- 每个企业的业务场景不同 - 设备监控的字段不适用于供应商管理
- 字段名称、规则逻辑需要定制 - "设备温度"对你的业务可能毫无意义
- 通过对话，我能帮你思考遗漏的地方 - 直接照抄会漏掉你业务的特殊需求
- **照抄例子 = 得到一个"看起来对"但"实际上错"的配置**
```

**效果**: ✅ Counter Rationalization #2 "我可以直接看例子然后自己写吗？"

### 改进3: 为什么Function不能跳过？(添加位置: 阶段三开始)

```markdown
### 为什么Function不能跳过？

**没有Function的后果**:
- ❌ 风险需要人工判断 → 设备温度95°C，但没人发现，30分钟后才有人巡检注意到，设备已经损坏

**有了Function之后**:
- ✅ 系统自动判断风险 → 温度95°C时，Function立即判断"高风险"，触发自动化规则，2秒内停机

**Function是Automation的大脑**:
- 没有Function，Automation无法判断何时触发
```

**效果**: ✅ Counter Rationalization #3 "Function看起来不重要，我跳过了"
